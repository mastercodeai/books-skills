# SKILL.md — 精益创业方法论应用指南

## 场景路由

根据当前所处阶段，直接跳转对应工具组合：

| 场景 | 触发信号 | 核心工具 | 预期产出 |
|------|---------|---------|---------|
| **MVP设计** | 有新想法但未验证；准备启动新项目 | 工具1→工具2→工具4 | 1个可测试MVP + 核心假设清单 + 可执行指标体系 |
| **转型决策** | 指标停滞≥3轮实验；用户反馈指向根本假设错误 | 工具3(第三步)→工具6→工具1 | 转型/坚持决策 + 根因分析 + 新假设 |
| **增长引擎** | 产品已验证价值假设；需要规模化增长 | 工具5→工具4→工具3(第二步) | 引擎诊断 + 核心指标 + 优化实验计划 |
| **创新核算** | 有多轮实验数据；需要评估进展真实性 | 工具4→工具3(第一二步)→工具7 | 基准线报告 + 同期群分析 + 批量优化计划 |

**快速路由**：如果你只有一句话的描述——
- "我有个新想法" → **MVP设计**
- "产品做了但不增长" → **增长引擎**
- "做了很多实验没效果" → **转型决策**
- "不知道我们到底进步没有" → **创新核算**

## 适用场景
- 新产品/新业务从0到1验证
- 大企业内部创新团队管理
- 创业项目方向判断（转型 vs 坚持）
- 产品开发流程优化（从大批量转向小批量迭代）
- 增长策略制定与衡量体系搭建

---

## 核心工作流：开发—测量—认知循环

```
[想法/假设] → 开发MVP → 推向真实顾客 → 收集数据(可执行指标)
     ↑                                              ↓
     ← ← ← 经证实的认知 → → 决策：转型 or 坚持 ← ← ←
```

**执行顺序**（反向规划）：
1. 我需要知道什么？（明确假设）
2. 我需要衡量什么？（定义可执行指标）
3. 我需要开发什么？（构建最小实验）

## 实操工具箱

### 工具1：识别信念飞跃假设

**操作步骤**：
1. 列出商业计划中所有未经验证的假设
2. 筛选出最关键的两个：
   - **价值假设**：用户真的需要这个吗？愿意为什么付费？
   - **增长假设**：用户如何发现我们？如何带来新用户？
3. 用类比/反证框架检验：有什么已知案例支持/反驳这个假设？
4. 将假设量化为可测试的预测（如"10%访客会注册试用"）

### 工具2：设计最小化可行产品

**选择MVP类型的决策树**：

```
产品概念是否难以口头解释？
├── 是 → 视频式MVP（如Dropbox 3分钟演示视频）
└── 否 → 产品需要深度个性化服务？
    ├── 是 → 贵宾式MVP（CEO亲自服务1-10个客户）
    └── 否 → 技术实现复杂度高？
        ├── 是 → Wizard of Oz MVP（人工模拟后台）
        └── 否 → 冒烟测试（Landing Page + 预订按钮）
```

**MVP检查清单**：
- [ ] 能否用<1个月/<5000元完成？
- [ ] 是否直接测试了核心信念飞跃假设？
- [ ] 是否能收集到可量化的用户行为数据？
- [ ] 是否去掉了所有"对学习不必要"的功能？

### 工具3：创新核算框架

**第一步：建立基准线**
```
用MVP获取真实数据：
- 转化漏斗：注册→激活→留存→推荐→付费
- 为每个环节记录基准数值
```

**第二步：调整引擎**
```
每次产品改动前：
1. 明确假设："改X会导致指标Y提升Z%"
2. 执行对比测试（A/B测试）
3. 用同期群分析验证结果
4. 好的改动 = 能改善用户行为的改动
```

**第三步：转型还是坚持**
```
信号灯系统：
🟢 坚持：驱动指标持续向目标移动，每次实验比上次更有效
🟡 警告：指标停滞，优化投入回报递减
🔴 转型：连续多轮实验无法推动核心指标
```

### 工具4：三可指标设计

| 检验标准 | 问题 | 虚荣指标反例 | 可执行指标正例 |
|---------|------|------------|--------------|
| 可执行 | 是否显示清晰因果？ | "本月4万点击" | "新注册页A比旧版B转化率高12%" |
| 可使用 | 非技术人员能否理解？ | "PV/UV比值" | "本周新注册用户中35%完成了首次购买" |
| 可审查 | 数据来源是否可追溯？ | "第三方报告说市场增长" | "可抽查任意用户的完整行为轨迹" |

### 工具5：三种增长引擎诊断

```
诊断流程：
1. 你的用户主要通过什么方式带来新用户？
   ├── 口碑/使用中自然传播 → 病毒式引擎
   ├── 高留存，几乎不流失 → 黏着式引擎
   └── 广告/销售团队获客 → 付费式引擎

2. 测量对应核心指标：
   病毒式 → 病毒系数 = 每用户平均邀请数 × 邀请转化率
   黏着式 → 复合增长率 = 新客获取率 - 流失率
   付费式 → LTV vs CPA（生命周期价值 vs 每取得成本）

3. 判断：核心指标是否 > 盈亏平衡点？
   ├── 是 → 继续优化，准备扩大规模
   └── 否 → 聚焦改善该指标，或考虑引擎转型
```

### 工具6："五个为什么"根本原因分析

**模板**：
```
问题：[具体症状]
1. 为什么？ → [直接原因]     → 按比例投入：[最小修复]
2. 为什么？ → [技术/流程原因] → 按比例投入：[小改进]
3. 为什么？ → [系统原因]     → 按比例投入：[中等改进]
4. 为什么？ → [管理/制度原因] → 按比例投入：[较大改进]
5. 为什么？ → [根本人为原因] → 按比例投入：[制度性变革]
```

**规则**：
- 所有相关人员必须到场（包括发现者、修复者、决策者）
- 目标是系统改进，不是追责个人
- 第一次犯错要容忍，同样的错误不允许第二次

### 工具7：小批量实施

**从大批量转小批量的步骤**：
1. 识别当前最大的批量瓶颈（如：季度发布、年度规划）
2. 将批量减半（季度→双周，双周→每天）
3. 投资自动化工具（持续集成/部署、自动化测试）
4. 建立"安灯"机制：发现质量问题立即停止，调查根本原因
5. 观察循环周期时间变化，持续缩小批量

## 转型决策框架

### 何时触发转型讨论
- 连续3轮以上优化实验，核心指标无显著改善
- 增长引擎的边际回报递减
- 用户反馈指向产品假设的根本性错误
- 现金跑道中转型次数即将耗尽

### 转型前必做功课
1. 准备完整的同期群数据（非虚荣指标）
2. 整理所有用户访谈/观察记录
3. 明确哪些认知已验证、哪些被推翻
4. 基于现有认知提出新的战略假设
5. 设计新MVP验证新假设

### 转型类型速查

| 转型类型 | 信号 | 示例 |
|---------|------|------|
| 放大转型 | 小功能比主产品更受欢迎 | YouTube从约会网站→视频平台 |
| 缩小转型 | 大产品中只有部分被需要 | 拆分为多个垂直产品 |
| 客户细分转型 | 实际用户≠预期用户 | 产品吸引企业而非个人 |
| 需求转型 | 解决的不是真痛点 | 用户说"不错但不需要" |
| 渠道转型 | 分发方式不可行 | 从线下转线上或反之 |
| 技术转型 | 同一需求有更好技术路径 | Web转App，或反之 |
| 平台转型 | 从应用转平台或反之 | 从单产品→开放平台 |
| 价值获取转型 | 盈利模式不可持续 | 从广告→订阅→交易佣金 |
| 增长引擎转型 | 当前引擎见顶 | 病毒式→付费式 |
| 渠道+价值转型 | 同时改变交付和变现 | 全面商业模式重构 |

## 失败模式库

### 模式F1：伪精益——为迭代而迭代
**症状**：每周都在做A/B测试，但测的都是按钮颜色、文案措辞
**根因**：没有回到工具1检查核心假设，只在表面优化
**修复**：暂停所有微调实验，重新审视"信念飞跃假设"是否已被测试

### 模式F2：MVP膨胀——最小变成了"差不多完整"
**症状**：MVP开发周期从2周变成3个月，"加完这个功能就上线"
**根因**：缺乏明确的"够用"定义，团队对"最小"有分歧
**修复**：强制执行时间盒（time-box），到期必须上线，哪怕只有一个功能

### 模式F3：数据幻觉——有指标但无认知
**症状**：仪表盘很漂亮，但团队无法回答"我们学到了什么"
**根因**：指标没有与假设挂钩，收集了数据但没有形成经证实的认知
**修复**：每次实验前写下"如果看到X，则假设成立/被推翻"的判定规则

### 模式F4：转型拖延——在错误方向上越陷越深
**症状**：已连续6+轮实验无改善，但仍"再试试这个方案"
**根因**：沉没成本心理 + 没有预设止损规则
**修复**：在项目启动时就设定"转型触发条件"（如：3轮实验核心指标无>5%改善）

### 模式F5：增长幻觉——虚荣指标掩盖真实问题
**症状**：总用户数持续增长，但留存率和活跃度下降
**根因**：花钱买来了不匹配的用户，漏斗底部在漏水
**修复**：用同期群分析看留存曲线，先修漏斗底部再扩大获客

### 模式F6：规模化陷阱——在未验证阶段就扩张
**症状**：刚有了第一个付费用户就招10个人、租办公室
**根因**：把早期信号误读为规模化信号
**修复**：区分"价值验证"和"增长验证"，前者只证明有人需要，后者才证明能规模化

### 模式F7：客户开发缺失——在办公室里猜测用户需求
**症状**：产品逻辑合理但无人使用，团队在内部争论功能优先级
**根因**：缺乏系统性的客户访谈和观察
**修复**：每周至少进行3次用户对话，用"观察行为"替代"询问意见"

## 反模式识别与纠正

### 反模式A1：大爆炸发布
**表现**：花6个月开发完整产品，一次性推向市场
**精益替代**：分阶段发布，每阶段验证一个核心假设
**转换路径**：先识别最危险的假设→设计最小实验→在2周内获得第一个用户反馈

### 反模式A2：事后精益
**表现**：产品失败后说"早该做MVP"，但下次仍然从完整方案开始
**精益替代**：在项目立项时就嵌入精益循环
**转换路径**：把"假设验证计划"作为项目立项的必要文档

### 反模式A3：指标操控
**表现**：为了让数据好看而改变指标定义（如把"活跃"定义为"登录过一次"）
**精益替代**：指标定义一旦确定就锁定，只允许新增不允许修改
**转换路径**：建立指标变更审批流程，每次变更需说明旧指标为何误导

### 反模式A4：精益官僚化
**表现**：每个小改动都要写假设文档、做A/B测试、出分析报告
**精益替代**：根据风险级别区分处理方式——低风险直接做，高风险才完整循环
**转换路径**：建立风险分级标准（影响核心假设=高风险，其他=低风险）

### 反模式A5：用户反馈陷阱
**表现**：用户说什么就做什么，产品变成需求堆砌
**精益替代**：区分"用户想要的"和"用户需要的"，用行为数据验证
**转换路径**：用户反馈输入假设池→优先级排序→设计实验验证→只有行为数据支持才做

## 关键检查点

### 检查点CP1：项目启动时
- [ ] 价值假设已书面化且可证伪
- [ ] 增长假设已书面化且可证伪
- [ ] MVP类型已选定且有明确的"够用"标准
- [ ] 时间盒已设定（建议2-4周）
- [ ] 3-5个可执行指标已定义
- [ ] 转型触发条件已预设

### 检查点CP2：MVP上线后第1周
- [ ] 至少有10个早期使用者接触了产品
- [ ] 收集到了第一批行为数据（非问卷数据）
- [ ] 用户行为是否符合预期假设？记录偏差
- [ ] 是否需要调整MVP范围？（小调=坚持，大调=可能是转型信号）

### 检查点CP3：每月战略回顾
- [ ] 同期群分析是否显示趋势改善？
- [ ] 核心指标是否在向目标移动？
- [ ] 最近3次实验各自产出了什么认知？
- [ ] 增长引擎诊断：当前引擎是否有效？是否需要切换？
- [ ] 批量大小是否可以进一步缩小？
- [ ] 现金跑道还剩多少？还能支撑几轮实验？

### 检查点CP4：转型决策会
- [ ] 完整的同期群数据已准备（非虚荣指标）
- [ ] 所有实验记录已整理，认知状态已标注（验证/推翻/待定）
- [ ] 失败模式库已对照检查（是否有F1-F7中的模式？）
- [ ] 反模式已对照检查（是否有A1-A5中的表现？）
- [ ] 新假设已提出且比旧假设更有依据
- [ ] 新MVP设计已初步完成
- [ ] 团队对转型方向有共识

### 检查点CP5：规模化前
- [ ] 价值假设已被多次独立验证
- [ ] 增长引擎的核心指标持续高于盈亏平衡点
- [ ] 有明确的单位经济模型（LTV/CPA、病毒系数等）
- [ ] 批量已缩到可接受的最小值，循环周期稳定
- [ ] "五个为什么"机制已建立，团队文化以系统改进为导向

## 常见陷阱与对策

| 陷阱 | 症状 | 对策 |
|------|------|------|
| 虚荣指标自嗨 | "总用户突破10万！"但留存率没变 | 切换到同期群分析 |
| 完美主义拖延 | "再加一个功能就发布" | 设定MVP最大开发时间（如2周） |
| 分析瘫痪 | 无尽的市场调研和战略规划 | 设定分析截止日，转为实验验证 |
| "先做再看" | 没有清晰假设就动手 | 先写下最核心的2个可证伪假设 |
| 转型恐惧 | 明知不行却不敢改变 | 安排固定的"转型还是坚持"会议 |
| 大批量死亡螺旋 | 发布周期越来越长，问题越来越多 | 投资自动化，缩小批量到单件流 |
| 五大罪状 | 出问题就互相指责 | 用"五个为什么"聚焦系统改进 |
| 秘密开发 | 长期隔离用户做完美产品 | 尽早用MVP接触真实用户 |

## 快速启动清单

如果你正在启动一个新项目，按以下顺序行动：

1. **写下2个信念飞跃假设**（价值假设 + 增长假设）
2. **设计1个最简MVP**验证这两个假设（不超过2周开发）
3. **定义3-5个可执行指标**构成转化漏斗
4. **找到10个早期使用者**让他们使用MVP
5. **建立同期群追踪**，每周分析一次数据
6. **每2周做一次**"转型还是坚持"评估
7. **用"五个为什么"**处理每个重大问题
8. **逐步缩小批量**，缩短反馈循环时间
