# 精益创业 — 埃里克·莱斯 (Eric Ries)

> 核心命题：创业是一个在极端不确定性条件下的实验过程，用科学方法而非直觉来发现可持续的商业模式。

---

## 一、核心循环：Build-Measure-Learn

精益创业的全部方法论围绕一个反馈循环展开：

```
想法 → 构建(Build) → 产品 → 衡量(Measure) → 数据 → 学习(Learn) → 新想法 → ...
```

- **构建**：用最小成本把想法变成可测试的产品/功能
- **衡量**：用真实用户行为数据评估效果
- **学习**：从数据中提取认知，决定坚持还是转型

**关键原则**：把这个循环转得尽可能快。每一圈循环的目标不是发布产品，而是**学到东西**。

---

## 二、经验证的学习 (Validated Learning)

> "创业所做的一切，归根结底是为了找到可持续商业模式的路径。经验证的学习是衡量进展的严谨方法。"

- 传统里程碑（完成产品、上线、获得融资）不是真正的进展
- **真正的进展 = 经过实验证实的认知**
- 每一个功能、每一个决策都应该是一个可证伪的假设
- 如果一个功能上线后无人使用，那不是"失败"，而是"学到了这个假设是错的"

**实践要求**：区分"虚荣指标"和"可行动指标"。下载量、页面浏览量是虚荣指标；留存率、转化率、每用户收入才是可行动指标。

---

## 三、最小可行产品 (MVP)

MVP 是用最少 effort 拿到最大量经验证学习的产品版本。

### MVP 的形式
| 类型 | 说明 | 案例 |
|------|------|------|
| 视频 MVP | 用视频演示产品概念 | Dropbox 早期演示视频 |
| 贵人式 MVP | 人工代替自动化流程 | Zappos 创始人手动买鞋发货 |
| 登录页 MVP | 用广告/落地页测试需求 | 看点击率决定是否开发 |
| 单一功能 MVP | 只做核心功能 | 许多 SaaS 产品 |

**核心要点**：
- MVP 不是"烂产品"，是"刚好够验证核心假设的产品"
- 如果你不对 MVP 感到尴尬，说明你发布得太晚了
- MVP 的目标用户是**早期采纳者**，不是主流市场

## 四、飞跃式假设 (Leap-of-Faith Assumptions)

每个创业都建立在两个核心假设之上：

### 1. 价值假设 (Value Hypothesis)
- 产品是否真正为用户创造了价值？
- 用户是否真的想要这个东西？
- 测试方法：看用户是否自发回来使用（留存、推荐）

### 2. 增长假设 (Growth Hypothesis)
- 新用户如何发现这个产品？
- 增长模式是可自我持续的吗？
- 测试方法：看自然增长曲线和病毒系数

**行动指南**：在写商业计划书之前，先把这两个假设写下来，然后设计最便宜的实验去验证它们。

## 五、创新会计 (Innovation Accounting)

传统会计无法衡量创新项目的进展。创新会计提供一套替代框架：

### 三步走
1. **建立基线**：用 MVP 收集初始数据，了解现状（哪怕现状很差）
2. **调整引擎**：每次迭代优化一个可衡量的指标
3. **转型还是坚持**：当指标增长停滞时，做出艰难决定

### 可行动指标 vs 虚荣指标
| 虚荣指标 | 可行动指标 |
|----------|-----------|
| 总注册用户数 | 每周活跃用户 |
| 总收入 | 每用户平均收入 |
| 页面浏览量 | 点击转化率 |
| 媒体报道数 | 用户推荐率 |

### 同期群分析 (Cohort Analysis)
- 不要用累计数据，要用按时间段分组的同期群数据
- 每月新用户群是否比上月表现更好？这才是真正的问题

## 六、转型 (Pivot)

> "转型不是失败，而是一种结构化的方向调整，受经验证的学习所启发。"

### 转型的时机信号
- 创新会计的指标持续停滞
- 每次实验的改善幅度越来越小
- 团队对核心假设的信心下降

### 转型的类型
| 类型 | 说明 |
|------|------|
| 放大转型 | 聚焦原产品的一个功能成为全部 |
| 缩小转型 | 把单一功能扩展为完整平台 |
| 客户细分转型 | 目标用户变了 |
| 客户需求转型 | 解决的问题变了 |
| 平台转型 | 从应用变平台或反过来 |
| 商业架构转型 | 从高利润低销量变低利润高销量（或反过来） |
| 价值获取转型 | 收入模式变了 |
| 渠道转型 | 分发渠道变了 |
| 技术转型 | 用不同技术实现相同价值 |

**关键**：转型需要勇气，也需要纪律。每次转型都应该产生新的可验证假设。

---

## 七、小批量 vs 大批量

- **小批量**（单件流）比大批量更高效
- 一次做一件事做完 vs 一次做十件再一起交付
- 小批量 = 更快发现缺陷 = 更快反馈 = 更低浪费
- 这解释了为什么持续部署（每天发布多次）优于季度发布

---

## 八、自适应组织

### 五个为什么
- 遇到问题时连问五次"为什么"，找到根本原因
- 每一层"为什么"对应一个解决方案，投入与问题严重程度成正比
- 避免过度治理，也避免忽视系统性问题

### 创新型组织的三个职能
1. **资源分配**：为创新项目留出独立空间
2. **流程管理**：用精益方法而非传统项目管理
3. **人员文化**：容忍失败但不容忍不学习

---

## 九、关键思维转变

| 旧思维 | 精益思维 |
|--------|---------|
| 先计划再执行 | 先实验再决策 |
| 追求完美产品 | 追求最快学习速度 |
| 避免失败 | 从失败中学习 |
| 静态商业计划 | 动态商业模式画布 |
| 关注产出 | 关注成果（学习） |
| 大规模发布 | 持续小规模迭代 |
| 专家意见驱动 | 数据驱动 |

---

## 十、一句话总结

**不要问"我们能不能做这个"，要问"我们应不应该做这个"——而回答这个问题的唯一方式，是用真实的实验去验证。**
