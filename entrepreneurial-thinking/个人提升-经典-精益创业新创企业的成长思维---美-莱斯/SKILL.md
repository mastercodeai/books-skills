# SKILL.md — 精益创业方法论技能卡

## 技能名称
**精益创业实践** (Lean Startup Practice)

## 适用场景
- 新产品/新业务从 0 到 1 探索
- 存量业务中孵化创新项目
- 判断一个产品方向是否值得继续投入
- 团队从"按计划执行"转向"按假设实验"

---

## 场景路由

```
你当前处于哪个阶段？
│
├─ 【还没开始做】→ 路由到 §MVP 设计
│   "先别建，先验证"
│
├─ 【已发布，但没增长】→ 路由到 §增长引擎 + §创新会计
│   "是渠道问题还是产品问题？"
│
├─ 【数据验证失败/停滞】→ 路由到 §转型决策框架
│   "该坚持还是该转向？"
│
├─ 【数据验证成功，准备规模化】→ 路由到 §规模化检查清单
│   "验证≠规模化，还需确认增长引擎"
│
└─ 【不知道该干什么】→ 先执行 §假设提炼
    "连假设都没有，其他都免谈"
```

## 核心方法论工具箱

### 1. Build-Measure-Learn 循环
```
假设 → 最小构建 → 收集数据 → 验证/推翻 → 新假设
```
**使用条件**：任何需要验证不确定性的决策
**操作步骤**：
1. 写下你的核心假设（必须可证伪）
2. 确定"推翻假设"的量化标准
3. 用最低成本构建可测试的原型
4. 收集真实用户行为数据（不是问卷）
5. 对比预设标准，做出判断

**检查点 ✅**：
- [ ] 假设是否可证伪（能被数据推翻）？
- [ ] "推翻标准"是否在实验前就写好？
- [ ] 循环周期是否 ≤ 2 周？

### 2. MVP 设计
**使用条件**：不确定用户是否需要某个功能/产品
**选择策略**：
| 你的处境 | MVP 类型 | 示例 |
|---------|---------|------|
| 概念还没验证 | 落地页/视频 MVP | Dropbox 早期演示视频 |
| 需求存在但不确定实现方式 | 贵人式 MVP（手动执行） | Zappos 创始人去鞋店拍照上架 |
| 核心功能已确定但不确定扩展功能 | 单一功能 MVP | Instagram 只做滤镜分享 |
| 不确定用户愿不愿意付费 | 预售 MVP | Kickstarter 众筹 |

**关键原则**：MVP 只验证一个核心假设，不要贪多

**检查点 ✅**：
- [ ] MVP 是否只验证一个假设？
- [ ] 构建时间是否 ≤ 2 周？
- [ ] 是否有明确的成功/失败判定标准？

### 3. 假设提炼
**操作模板**：
```
假设：我们相信 [目标用户] 会因为 [原因] 使用/购买 [功能/产品]
验证标准：当 [量化指标] 达到 [阈值] 时，假设成立
实验方式：通过 [具体方法] 在 [时间] 内验证
成本：不超过 [资源上限]
```

**假设优先级排序**（按致命程度）：
1. **价值假设**：用户真的需要这个吗？（最先验证）
2. **增长假设**：用户会自发传播/持续使用吗？
3. **商业模式假设**：能赚到钱吗？
4. **技术假设**：我们做得到吗？

**检查点 ✅**：
- [ ] 是否已识别最致命的假设排第一？
- [ ] 假设是否足够具体（不是"用户会喜欢"这种废话）？

### 4. 创新会计
**使用条件**：需要衡量创新项目的进展（传统 KPI 失效时）
**操作步骤**：
1. 建立基线：当前关键指标是多少？
2. 设定改善目标：每次迭代预期改善多少？
3. 同期群分析：按时间段分组对比，看趋势
4. 决策点：连续 N 次迭代无显著改善 → 考虑转型

**检查点 ✅**：
- [ ] 是否使用同期群分析（而非累计数据）？
- [ ] 是否有预设的"转型触发阈值"？
- [ ] 是否每周复盘指标趋势？

### 5. 转型决策框架
**使用条件**：当前方向验证失败或增长停滞
**决策清单**：
- [ ] 创新会计数据是否显示增长停滞？
- [ ] 是否已尝试足够多次迭代（通常 ≥ 3-5 次）？
- [ ] 失败原因是假设错误还是执行不力？
- [ ] 转型方向是否有新的可验证假设？
- [ ] 转型后的实验计划是否已制定？

**转型类型速查**：
| 类型 | 触发信号 | 示例 |
|------|---------|------|
| 客户细分转型 | 用户来了但不是预期那群人 | YouTube 从约会网站→视频分享 |
| 需求转型 | 用户喜欢产品但用途不同 | Slack 从游戏→企业通讯 |
| 渠道转型 | 产品好但获客方式不对 | 从线上转线下直销 |
| 商业模式转型 | 用户多但不赚钱 | 从付费转免费增值 |
| 技术转型 | 功能对但实现方式太贵 | 用更低成本技术替代 |

### 6. 增长引擎
**使用条件**：MVP 已验证价值假设，需要找到可持续增长方式

**三种增长引擎**（书中核心框架）：

| 引擎类型 | 驱动力 | 关键指标 | 适用场景 |
|---------|--------|---------|---------|
| **黏性增长** | 留存率 > 流失率 | 留存率、流失率 | SaaS、社区类产品 |
| **病毒式增长** | 用户自发传播 | 病毒系数 K > 1 | 社交、工具类产品 |
| **付费增长** | LTV > CAC | LTV/CAC ≥ 3 | 电商、B2B |

**检查点 ✅**：
- [ ] 你的主力增长引擎是哪一种？（不能同时追求三种）
- [ ] 黏性引擎：流失率 < 新增率？
- [ ] 病毒引擎：K 系数是否 > 1？
- [ ] 付费引擎：LTV/CAC 是否 ≥ 3？

## 失败模式诊断表

| 症状 | 可能根因 | 诊断动作 | 修复方向 |
|------|---------|---------|---------|
| 做了很多但没增长 | 增长假设未验证 | 检查病毒系数和留存曲线 | 重新验证增长引擎 |
| 用户说好但不付费 | 价值假设验证不足 | 区分"口头喜欢"和"实际付费" | 用预售/付费墙验证真需求 |
| 留存率持续下降 | 核心价值未被感知 | 做用户流失访谈 | 强化 Aha Moment 引导 |
| 指标好看但团队迷茫 | 虚荣指标陷阱 | 检查是否用了可行动指标 | 切换到同期群分析 |
| 每次迭代都改善一点但无质变 | 可能在错误方向上微调 | 检查是否在优化错误假设 | 考虑结构性转型 |
| MVP 发布后没人用 | 问题不够痛或渠道错误 | 区分"没人知道"和"知道但不要" | 分别解决渠道/价值问题 |

## 反模式与解药

| 反模式 | 症状 | 解药 |
|--------|------|------|
| 构建者陷阱 | 花6个月做"完美"产品再发布 | 强制2周内出MVP |
| 虚荣指标驱动 | "我们有10万用户！"但留存率2% | 只看可行动指标：留存、活跃、付费 |
| 假精益 | 声称在做MVP但拒绝发布不完美的东西 | "如果你不尴尬，说明发布太晚" |
| 没有假设 | "先做了看看" | 每次实验前写假设卡片 |
| 拒绝转型 | 已验证失败仍坚持 | 设定自动触发转型的阈值 |
| 过度转型 | 一遇挫折就换方向 | 要求 ≥ 3 次迭代后才能评估转型 |
| 数据崇拜 | 只看数据不听用户 | 定性访谈 + 定量数据双轨并行 |
| MVP 膨胀症 | "再加一个功能就发布" | MVP 清单：只能有一个核心功能 |

## 加速技巧

1. **缩短循环**：目标是天级而不是月级的 Build-Measure-Learn 循环
2. **A/B 测试优先**：能做 A/B 就不做大版本发布
3. **定性 + 定量结合**：数据告诉你"是什么"，用户访谈告诉你"为什么"
4. **会计账本法**：每个实验记录：假设、实验、结果、学习，形成团队知识库
5. **五为什么根因分析**：出了问题连问五次为什么，找到根本原因并分级修复
6. **同期群思维**：永远按用户获取时间分组看数据，累计数据会掩盖真相

## 规模化检查清单

当 MVP 验证通过，准备放大投入前：

- [ ] 价值假设已通过真实付费/留存数据验证
- [ ] 已确认主力增长引擎类型且指标健康
- [ ] LTV/CAC ≥ 3（如适用）
- [ ] 核心流程已文档化，新人可复制
- [ ] 已识别规模化后的最大风险并有预案

## 一句话使用指南
> 面对不确定性时，不要争论谁对谁错——设计一个最便宜的实验来验证，用数据说话。

## 实战案例库

### 案例 1：Dropbox — 视频 MVP 验证价值假设
- **背景**：创始人 Drew Houston 不确定"文件同步"是否有真实需求
- **实验**：发布 3 分钟演示视频，一天内注册量从 5K → 75K
- **学习**：价值假设通过；技术假设（能否实现）仍需验证
- **关键点**：视频 MVP 成本极低，但只验证了"用户想要"，不验证"用户会用"

### 案例 2：Zappos — 贵人式 MVP 验证购买意愿
- **背景**：不确定人们是否愿意在线买鞋（不能试穿）
- **实验**：创始人去鞋店拍照上架，有人下单就去买回来发货
- **学习**：即使不能试穿，用户仍愿意在线购买；退货率可控
- **关键点**：手动执行验证需求，避免先建仓库和物流系统

### 案例 3：Airbnb — 假设验证中的关键转折
- **背景**：早期增长缓慢，团队不确定问题出在哪
- **实验**：创始人亲自去纽约房东家拍照，替换业余照片
- **学习**：问题不是"没人要住"而是"照片太丑导致不信任"；收入翻倍
- **关键点**：增长停滞时，可能是体验问题而非价值问题；定性观察比数据更有洞察

### 案例 4：Buffer — 落地页验证定价意愿
- **背景**：不确定用户是否愿意为社交媒体管理工具付费
- **实验**：搭建落地页展示定价方案，观察用户是否点击"注册付费"
- **学习**：点击率足够高，验证了付费意愿；随后才开始开发
- **关键点**：定价假设要尽早验证，不要等到产品做好再想变现

### 案例 5：Food on the Table — 极致手工 MVP
- **背景**：想做个性化食谱+购物清单推荐
- **实验**：创始人手动为第一个用户做食谱推荐和购物清单，每周亲自服务
- **学习**：手动服务揭示了用户真正关心的是"省时间"而非"新菜式"
- **关键点**：贵人式 MVP 能揭示你用自动化无法发现的用户需求

## 常见场景决策树（补充）

### 场景 A：团队说"我们已经验证过了"
→ 追问：
1. 验证标准是什么？谁定的？实验前还是实验后定的？
2. 用的是同期群数据还是累计数据？
3. 有多少用户真的付费/留存，而不只是注册？
→ 如果回答模糊 → 重新验证，这次先写假设卡片

### 场景 B：老板说"别验证了，直接做"
→ 应对：
1. 用数据类比：同领域 N 个创业公司，M 个在同类方向失败
2. 提出"2 周验证计划"：最小成本说服老板
3. 如果老板仍拒绝 → 记录假设，3 个月后回看结果

### 场景 C：MVP 数据好但用户增长不起来
→ 诊断路径：
1. 病毒系数 K < 1？→ 产品没有自然传播力，需加入分享机制或口碑触发点
2. 留存曲线平但新增少？→ 渠道问题，不是产品问题
3. 留存曲线持续下降？→ 产品核心价值未被感知，做流失用户访谈

### 场景 D：已转型 3 次仍无突破
→ 核查：
1. 每次转型是否有新的可验证假设？还是"拍脑袋换方向"？
2. 是否在同一个用户群中打转？→ 考虑换客户细分
3. 团队是否还有热情？→ 有时候问题不是方向而是团队状态

## 补充失败模式

| 症状 | 可能根因 | 诊断动作 | 修复方向 |
|------|---------|---------|---------|
| 团队对"验证结果"有争议 | 假设不够具体，标准模糊 | 回查实验前写的假设卡片 | 重做实验，这次标准必须量化 |
| 每次迭代都像重新开始 | 没有形成知识库 | 检查是否有实验记录 | 建立"假设-实验-结果-学习"日志 |
| 用户增长快但口碑差 | 虚荣增长（补贴/刷量） | 检查自然推荐率和 NPS | 停止补贴，看真实留存 |
| MVP 发布后获得媒体关注但没留存 | 曝光不等于价值验证 | 分析注册→激活→留存漏斗 | 关注激活率，而非注册数 |
| "对标竞品"变成主要策略 | 战略懒惰 | 问"竞品验证过什么？我们的用户有何不同？" | 回到自己的假设验证 |
| 创始人和产品经理意见不一致 | 假设未对齐 | 各自写下核心假设，对比差异 | 先对齐假设，再讨论方案 |

## 补充检查点

### 假设提炼阶段
- [ ] 假设是否只包含一个变量？（不能同时验证需求+技术+商业模式）
- [ ] 是否区分了"必须为真"和"最好为真"的假设？
- [ ] 实验是否排除了混淆变量？（价格、渠道、时机等因素）

### Build-Measure-Learn 执行中
- [ ] 是否在实验开始前就写下了"推翻标准"？
- [ ] 收集的数据是行为数据还是态度数据？（行为 > 态度）
- [ ] 是否设置了实验的最大时间窗口？（防止无限期"再看看"）

### 转型决策时
- [ ] 是否排除了"执行不力"的干扰？（方向对但没做好 ≠ 方向错了）
- [ ] 转型方向是否有至少一个可验证假设？
- [ ] 是否已通知所有利益相关方并获得共识？

### 规模化后
- [ ] 增长引擎是否还健康？（每月复查 LTV/CAC、留存率、K 系数）
- [ ] 是否建立了持续实验的文化？（规模化后仍需创新）
- [ ] 核心指标是否被团队所有人理解？（不只是管理层）
