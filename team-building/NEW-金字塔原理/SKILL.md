## 技能一：结论先行，构建金字塔表达

**场景**：撰写报告、邮件、方案，或进行口头汇报时

**操作步骤**：
1. 先写出你要表达的**一个**核心结论/中心思想
2. 问自己："读者听到这个结论后会问什么？"→ 写出3~5个回答（关键句要点）
3. 对每个关键句要点，再问"为什么/如何做到"→ 继续向下展开
4. 每组要点不超过5个，超过则重新归类分组
5. 表达时从金字塔顶端开始，结论先行

**检查清单**：
- [ ] 文章是否只有一个中心思想？
- [ ] 每个层次的思想是否是对下一层次的概括？
- [ ] 同组思想是否属于同一逻辑范畴？
- [ ] 同组思想是否有明确的逻辑顺序？

**常见失败模式**：
- ❌ **流水账**：按时间顺序罗列过程，结论藏在最后一页 → 改为结论前置
- ❌ **多中心**：一篇文章塞入2~3个不相关的主题 → 拆分为多篇
- ❌ **层次错乱**：关键句要点是行动，支撑论据却变成了新的结论 → 检查每层是否是"概括→展开"关系

**反模式警告**：
> 🚫 "先铺垫背景再慢慢引出结论" — 金字塔原理的核心是**结论先行**，背景放在序言中，不是正文开头。读者没有耐心等你铺垫。

## 技能二：SCQA序言框架

**场景**：写任何文章、报告、邮件的开头；做演讲的开场白

**模板**：

```
背景(S)：[读者已知的、不会引起争议的事实]
冲突(C)：[发生了什么导致问题/变化]
疑问(Q)：[读者自然会问的问题]
答案(A)：[你的核心观点/建议]
```

**四种变体**：
| 风格 | 顺序 | 效果 |
|------|------|------|
| 标准式 | S→C→A | 平稳、客观 |
| 开门见山式 | A→S→C | 直接、高效 |
| 突出忧虑式 | C→S→A | 强调紧迫性 |
| 突出信心式 | Q→S→C→A | 激发好奇心 |

**示例**（改写自书中）：
> S：大客户建议改变记账方式。C：我们需要审核该方案是否可行。Q：建议是否可行？A：可行。原因有三：提供所需信息、增加现金流、减少工作量。

**常见失败模式**：
- ❌ **冲突不够**：背景写了一大段，冲突轻描淡写 → 冲突必须能引发读者的**真实疑问**
- ❌ **答案太弱**：序言的答案和正文结论不一致 → A必须与金字塔顶层完全吻合
- ❌ **假背景**：S中夹杂了观点而非事实 → 背景必须是读者会认同的已知信息

**反模式警告**：
> 🚫 把序言写成"感谢领导、回顾历史"式的客套话 — SCQA的S是为C和Q服务的，不是装饰。

## 技能三：MECE分类与逻辑分组

**场景**：分析问题、归纳原因、设计方案、分配任务

**操作步骤**：
1. 列出所有相关的思想/事实/因素
2. 找出它们的**共同属性**，确定分组标准
3. 按分组标准归类，确保：
   - **相互独立（ME）**：各组之间无重叠
   - **完全穷尽（CE）**：所有内容都有归属，无遗漏
4. 用一个**单一名词**命名每组（如"原因"、"步骤"、"问题"）
5. 组内按逻辑顺序排列（时间/结构/重要性）

**自检方法**：
- 能否用一个名词概括这组所有思想？
- 是否存在既属于A组又属于B组的项？
- 是否有遗漏未归类的项？
- 组内是否有明确的排列理由？

**常见失败模式**：
- ❌ **假MECE**：用了"内部/外部"分类，但有些项同时属于内外 → 分类标准必须**互斥**
- ❌ **遗漏型**：只列了3个原因，实际有5个 → 用"还有其他吗？"追问自己
- ❌ **层级混淆**：把"成本"和"人力成本"放在同一层级 → 上层应为"成本"，下层才细分

**反模式警告**：
> 🚫 "先列10个点，再强行分成3组" — 正确做法是先确定分类维度，再填充内容。强行归类会导致逻辑混乱。

## 技能四：自下而上思考法

**场景**：思路不清晰，无法确定主题或结论时

**三步法**：
1. **列出要点**：把所有想表达的思想罗列出来
2. **找出关系**：分析各要点之间的因果、并列、时间等逻辑关系
3. **得出结论**：从各组要点中提炼出概括性结论

**实操示例**：
```
要点：
- 排版效率低于平均水平20%~50%
- 每项任务都用相同工序
- 工资低于同行 → 员工流失 → 加班过多

分析：存在两条因果链
→ 结论：可通过简化工序 + 提高工资来降低成本
```

**常见失败模式**：
- ❌ **停在第二步**：找出了关系但没提炼结论 → 结论必须回答"所以呢？"
- ❌ **强行因果**：把相关性当因果性 → 检查每个因果链是否有"如果A则必然B"的关系

**反模式警告**：
> 🚫 列出20个要点后不做归纳就直接输出 — 要点太多说明你还没想清楚，继续归类直到每组≤5个。

## 技能五：逻辑顺序选择

**场景**：组织同一组内的多个要点时

| 顺序类型 | 判断标准 | 举例 |
|----------|----------|------|
| 时间顺序 | 前后有因果/先后关系 | 步骤1→步骤2→步骤3 |
| 结构顺序 | 属于整体的不同部分 | 东部→中部→西部 |
| 程度顺序 | 具有共同属性但程度不同 | 最重要→次重要→再次要 |

**选择方法**：
- 如果是行动/步骤 → 时间顺序
- 如果是对结构的描述 → 结构顺序
- 如果是对某共同属性的分类 → 程度顺序

**检查点**：
- [ ] 组内所有要点是否使用**同一种**顺序？
- [ ] 顺序是否符合读者的直觉预期？
- [ ] 是否存在"混搭"（时间+程度混在一起）？

## 技能六：概括总结句的写法

**场景**：为一组思想写总结/主题句/标题

**行动性思想**（做某事）：
- ❌ "加强地区作用" → 模糊
- ✅ "赋予各地区制订计划的权力" → 明确
- **原则**：概括语必须说明采取行动后取得的**具体结果**

**描述性思想**（说某事）：
- ❌ "存在3个问题" → 缺乏思想
- ✅ "主要问题是缺乏授权" → 揭示本质
- **原则**：概括语必须说明这些思想共同点的**含义/意义**

**三个层次的概括**：
1. 找出结构上的共性（主语/谓语/隐含意义的共同点）
2. 寻找更密切的联系
3. 完成归纳跃进，概括出主题思想

**常见失败模式**：
- ❌ **空壳句**："关于XX的若干建议" → 没有任何信息量
- ❌ **只列数量**："有3个原因" → 读者想知道的是原因是什么，不是有几个
- ❌ **概括过宽**：支撑材料讲的是"审批流程"，概括句写成了"管理问题" → 概括必须精确

## 技能七：界定问题框架

**场景**：面对复杂问题，需要理清问题本质时

**四要素展开**：

```
切入点/序幕 → 困扰/困惑 → R1（现状/非期望结果）
                              ↓
                           R2（目标/期望结果）
```

**五步分析**：
1. 是否有问题/机会？（判断差距是否存在）
2. 问题在哪里？（定位具体环节）
3. 为什么存在？（寻找根本原因）
4. 我们能做什么？（列出可能方案）
5. 我们应该做什么？（选择最优方案）

**将问题转化为序言**：
- S = 背景（切入点 + 正常状态）
- C = 困扰 + R1 + R2
- Q = 如何从R1到R2？
- A = 你的方案

**检查点**：
- [ ] R1和R2是否描述得足够具体（可量化）？
- [ ] 困扰是否有明确的触发事件？
- [ ] 方案是否直接回应"如何从R1到R2"？

**常见失败模式**：
- ❌ **R1和R2不清晰**：R2写了"改善现状" → 必须写成具体的期望状态
- ❌ **跳过步骤3**：直接从问题跳到方案 → 没找根因，方案治标不治本

## 技能八：演绎推理 vs 归纳推理的选择

**场景**：组织同一层次的思想时，选择横向逻辑关系

**何时用演绎**：
- 读者需要先理解原理才能接受结论
- 你的结论与读者预期不同，需要解释"为什么"
- 推理链不超过4步

**何时用归纳**：
- 直接回答"如何做"
- 各要点具有明显共性
- 读者更关心行动方案而非原理

**关键原则**：在关键句层次上，**优先使用归纳法**，因为更便于读者理解。将演绎推理放在金字塔较低层次。

**常见失败模式**：
- ❌ **演绎过长**：推理链超过4步 → 读者会迷失，应拆分或改用归纳
- ❌ **假归纳**：把演绎伪装成归纳（先给结论再编理由） → 归纳要求各要点确实独立成立

**反模式警告**：
> 🚫 关键句层次全部用演绎 — 大段推理会让听众走神。先用归纳给出"是什么"，再在支撑层用演绎解释"为什么"。

## 技能九：PPT演示文稿设计

**场景**：制作向多人演示的幻灯片

**比例原则**：图表90% + 文字10%

**文字幻灯片**：
- 每张只说明一个论点
- 使用完整陈述句（非标题性语言）
- 不超过6行或30个单词
- 字号计算：观众距离(英尺) ÷ 32 = 最小字号(英寸)

**图表幻灯片**：
- 标题直接传递信息（用含动词的完整句子）
- 选择最能表现论点的图表类型
- 信息尽量简单易懂

**故事梗概制作流程**：
1. 写详细序言
2. 按金字塔结构写故事梗概
3. 决定每张的呈现方式（文字/图表）
4. 准备讲稿
5. 设计绘图
6. 排练

**常见失败模式**：
- ❌ **文字堆砌**：一张PPT塞满文字 → 每张只传达一个观点
- ❌ **无故事线**：各页独立，没有起承转合 → 先写梗概再做PPT
- ❌ **标题无信息**：标题写"数据分析" → 改为"销量下降20%，主因是渠道A衰退"

**反模式警告**：
> 🚫 先做PPT再想结构 — 必须先有金字塔结构的故事梗概，再决定每页放什么。

## 技能十：标题与格式呈现

**场景**：撰写长篇文章、报告时

**多级标题法**：
- 不同层次用不同标记区分
- 层次越低越靠右
- 同层次用同一格式
- 同组标题用相同句型（对仗）

**标题六条规则**：
1. 每一层级不可能只有一个标题
2. 相同思想使用相同句型
3. 标题提炼思想精髓
4. 标题与正文分开考虑
5. 每组标题提前集中介绍
6. 不要滥用标题

**过渡方法**：
- 讲故事：用背景→冲突→疑问引出下一节
- 承上启下：从前一节选关键词用于下一节开头
- 总结：长章节结束时完整复述主要论点

**检查点**：
- [ ] 只看标题能否理解全文逻辑？
- [ ] 同层级标题是否句式对仗？
- [ ] 标题是否传递了实质性信息（而非"概述""分析"这类空词）？

## 全局自检清单（每次使用后过一遍）

1. **单一性**：整篇是否只有一个核心思想？
2. **金字塔结构**：上层是否概括下层？同层是否逻辑并列？
3. **MECE**：同组要素是否互斥且穷尽？
4. **序言完整**：是否有SCQA？冲突是否足以引发疑问？
5. **概括精准**：总结句是否揭示了思想的实质含义？
6. **顺序一致**：同组内是否只使用一种逻辑顺序？
7. **读者导向**：是否从读者已知出发，回答读者的疑问？
