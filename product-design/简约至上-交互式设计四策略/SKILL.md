# SKILL: 简约至上 — 交互式设计四策略

> 来源: Giles Colborne《简约至上：交互式设计四策略》
> 适用场景: 产品设计、UI/UX设计、功能规划、交互设计、跨设备体验

---

## 场景路由（Scene Router）

根据输入关键词，快速定位到对应策略区域：

```
输入 → 匹配场景 → 推荐策略
```

| 关键词/场景 | 路由目标 | 说明 |
|------------|---------|------|
| 太复杂、功能多、界面乱、臃肿 | 删除（Delete） | 砍掉不必要的功能和视觉噪音 |
| 找不到、迷路、分类混乱、导航差 | 组织（Organize） | 围绕用户行为重新分组和布局 |
| 太拥挤、选项多、界面密、信息过载 | 隐藏（Hide） | 渐进展示，只露核心 |
| 跨平台、多设备、手机放不下、角色分工 | 转移（Displace） | 设备间/人机间重新分配任务 |
| 用户抱怨、为什么没人用、数据差 | 用户洞察清单 | 先做用户研究再动刀 |
| 不确定用哪个策略 | 快速诊断表 | 按症状对号入座 |
| 设计评审、自查、验收 | 检查点清单 | 逐项检查设计质量 |
| 为什么会翻车 | 失败模式 | 对照排查根因 |

**路由规则：**
1. 优先识别用户的**核心痛点**而非表面症状
2. 痛点不明确时 → 先跑用户洞察清单
3. 多个痛点共存 → 按「删除 → 组织 → 隐藏 → 转移」顺序处理
4. 涉及跨设备/多人协作 → 转移策略优先考虑

---

## 何时调用

- 设计或简化产品界面和交互
- 决定哪些功能保留、哪些删除
- 评估产品复杂度是否超标
- 需要在功能丰富与易用性之间做取舍
- 面对"功能堆砌"或"用户抱怨太复杂"的问题
- 跨设备/跨平台的产品规划
- 产品改版或重构交互流程

## 四策略决策框架

### 1. 删除（Delete）— 最激进，效果最大

**什么时候用：** 功能过多、界面混乱、用户找不到核心功能

**操作清单：**
- 问"为什么要留着它"而非"为什么要去掉它"
- 识别并砍掉残缺/半生不熟的功能（沉没成本误区）
- 消灭"假如用户想……"的假设性需求
- 关注用户目标而非流程——问"还有其他解决方式吗"
- 清除视觉混乱：空白替代线条、减少强调、最多3个层次
- 砍掉多余文字：引见性文字、不必要说明、繁琐解释
- 用聪明的默认值减少选择
- 设计不出错的界面 > 设计好的错误消息
- 在结账/关键决策页面去掉导航链接（亚马逊做法）

**红线：** 不能删除用户的控制感（苹果无按钮电梯反面教材）

### 2. 组织（Organize）— 最常见，投入最小

**什么时候用：** 功能合理但分布混乱，用户找不到想用的

**操作清单：**
- 只强调1-2个最重要的起点（Flip的红色录制键）
- 围绕用户行为而非内部逻辑来组织
- 分块数量尽量少（≤6组）
- 用网格对齐界面元素（不可见网格也有效）
- 重要元素放大，不重要元素缩小（重要性1/2→大小1/4）
- 相似元素放在一起，减少色标/标签/边框
- 用感知分层（颜色/灰度/大小）处理复杂信息
- 观察用户的"期望路径"而非只看设计稿

**陷阱：** 字母表排序（不知道叫什么就找不到）、按格式分类（打断关联）

### 3. 隐藏（Hide）— 最灵活，需谨慎

**什么时候用：** 功能不常用但不能删，需要保持界面清爽

**操作清单：**
- 隐藏一次性设置、偏好选项、地区信息
- 用渐进展示：核心功能默认可见，专家功能展开显示
- 用阶段展示：随用户深入逐步展示（讲故事式展开）
- 适时出现：功能在用户需要的精确时刻和位置出现（纽约时报词典）
- 标签放在用户关注点上，比做大更重要
- 用"邀请探索"（小三角图标）替代"高级"标签

**红线：**
- 不要让用户自定义界面（主流用户不会、也不想）
- 不要自动定制（Office 2000自适应菜单的失败）
- 不要让隐藏成为删除的拖延借口

### 4. 转移（Displace）— 最巧妙，需要系统思维

**什么时候用：** 单一平台承载不了全部功能，需要跨设备/跨角色分配

**操作清单：**
- 设备间转移：手机负责采集/轻交互，桌面负责分析/展示
- 向用户转移：让计算机管存储和计算，让人管决策和估算
- 创建开放式体验：一个工具多种用途（菜刀/钢琴模型）
- 让用户用自然方式输入，计算机负责结构化（非结构化数据）
- 信任用户能完成任务，不要把用户当孩子

**核心表格——人机分工：**

| 人擅长 | 计算机擅长 |
|--------|-----------|
| 设定目标、规划 | 执行、存储 |
| 估算、辨别 | 精确计算 |
| 想象、做预算 | 交叉引用、排序 |

**跨设备分配速查：**

| 设备 | 适合承担 | 不适合承担 |
|------|---------|-----------|
| 手机 | 快速录入、通知、轻审批 | 复杂报表、深度编辑 |
| 平板 | 展示、手绘、阅读 | 精细数据输入 |
| 桌面 | 深度分析、批量操作、复杂配置 | 即时场景、移动采集 |

## 用户洞察清单

1. **为主流用户设计**，忽略专家型用户（iPod教训）
2. **走出办公室**观察真实环境中的用户（干扰、中断、压力）
3. 用户需要**掌控感**：掌控技术 + 掌控生活
4. 用"然后呢？"追问法挖掘深层感情需求
5. 讲故事描述体验：环境 + 角色 + 情节
6. 设定**极端可用性目标**（瞬间响应、毫不费力、不出错）
7. 反复分享你的认识，直到自己都厌烦

## 快速诊断表

| 症状 | 对应策略 | 典型做法 |
|------|---------|---------|
| 用户说"太复杂" | 删除 | 砍掉64%未使用的功能 |
| 用户说"找不到" | 组织 | 围绕行为分块，网格对齐 |
| 界面"太拥挤" | 隐藏 | 渐进展示 + 适时出现 |
| 单一设备承载不了 | 转移 | 手机采集 + 桌面分析 |
| 用户说"害怕弄坏" | 删除+隐藏 | 设计不出错界面，隐藏危险操作 |
| 用户在压力下出错 | 删除 | 极端目标：不出错 |
| 用户说"功能不够" | 先洞察再决策 | 可能是组织/隐藏问题，不一定是真缺功能 |
| 多角色/多设备协作混乱 | 转移 | 明确分工边界，设备专精化 |

## 失败模式（Failure Modes）

### 删除失败
| 失败信号 | 根因 | 修复 |
|---------|------|------|
| 用户投诉功能消失 | 砍了高频使用的功能 | 用数据验证使用频率再删 |
| 团队抵制删功能 | 没有共识和数据支撑 | 先做使用频率分析，用数据说话 |
| 删完界面仍乱 | 只删功能没清视觉噪音 | 同步清理布局、间距、层次 |

### 组织失败
| 失败信号 | 根因 | 修复 |
|---------|------|------|
| 用户仍说找不到 | 按内部逻辑分组而非用户行为 | 用卡片分类测试验证分组合理性 |
| 导航层级太深 | 分组粒度太细 | 合并小类，控制在2-3层内 |
| 用户选错类别 | 标签用语不是用户语言 | 用用户原话做标签 |

### 隐藏失败
| 失败信号 | 根因 | 修复 |
|---------|------|------|
| 用户不知道某功能存在 | 隐藏太深，缺乏线索 | 在相关场景加"邀请探索"提示 |
| 设置页变成垃圾场 | 所有东西都往设置里塞 | 区分一次性设置和常用设置 |
| 用户发现功能后失望 | 隐藏的是核心功能 | 核心功能不应隐藏，重新评估优先级 |

### 转移失败
| 失败信号 | 根因 | 修复 |
|---------|------|------|
| 用户在设备间重复操作 | 状态没同步 | 确保云端状态实时同步 |
| 手机端操作太重 | 转移时没考虑设备限制 | 手机只做轻操作，重活留给桌面 |
| 用户不知道该用哪个设备 | 缺乏引导 | 在关键节点给出设备推荐提示 |

## 反模式（Anti-Patterns）

| 反模式 | 表现 | 正确做法 |
|--------|------|---------|
| ❌ 假设性需求 | "假如用户想……"就加功能 | 只做目标用户**经常**遇到的 |
| ❌ 功能军备竞赛 | 功能越多越好 | 试用后用户更看重可用性 |
| ❌ 设计甩锅 | 让用户自定义界面 | 设计师要有主见 |
| ❌ 自动定制 | 界面自动适应用户行为 | 用户会困惑（Office 2000教训） |
| ❌ 标签歧视 | 用"高级/专家"标签 | 让用户觉得不配 → 直接隐藏即可 |
| ❌ 说明书依赖 | 靠文档解决问题 | 界面本身要自解释 |
| ❌ 长向导崇拜 | 用wizard简化一切 | 向导越长体验越差 |
| ❌ 难度删除 | 只删难实现的功能 | 聚焦用户价值，而非开发难度 |
| ❌ 沉没成本保留 | "都开发了就留着吧" | 功能成本≠用户价值 |
| ❌ 数据洁癖 | 为了简洁删除所有数据 | 保留用户决策所需的关键信息 |

## 检查点清单（Checkpoints）

### 设计前
- [ ] 明确了目标用户是谁（不是"所有人"）
- [ ] 知道用户最常做的3件事
- [ ] 收集了真实用户反馈/数据
- [ ] 确定了"什么绝对不能删"

### 设计中
- [ ] 主路径（最常见任务）是否最突出
- [ ] 视觉层次是否清晰（≤3层）
- [ ] 分组逻辑是否基于用户行为
- [ ] 隐藏功能是否有发现线索
- [ ] 是否存在"假如用户想……"的陷阱
- [ ] 默认值是否合理
- [ ] 跨设备场景是否考虑了（如适用）

### 设计后/评审
- [ ] 核心任务能否在3步内完成
- [ ] 新用户能否无指导完成首次任务
- [ ] 老用户是否能找到进阶功能
- [ ] 是否通过了"5秒测试"（5秒内理解页面目的）
- [ ] 对照反模式清单逐项排查
- [ ] 删除的功能是否有数据支撑

## 实战案例库

### 案例 1：删除 — Google 首页 vs Yahoo 首页
- **背景**：2000 年代，Yahoo 首页塞满链接和功能，Google 只有一个搜索框
- **策略**：删除一切非核心功能，聚焦"搜索"这一个用户目标
- **结果**：Google 首页加载更快、用户任务完成率更高、品牌认知更清晰
- **关键点**：删除不是"做减法"，是"只做用户真正需要的那一件事"

### 案例 2：组织 — Flip 相机的红色录制键
- **背景**：传统摄像机按钮密密麻麻，用户不知道从哪开始
- **策略**：只强调一个入口——大红色录制键，其他功能全部弱化
- **结果**：用户上手时间从 15 分钟降到 < 1 分钟
- **关键点**：组织的核心是"让用户一眼看到最重要的起点"

### 案例 3：隐藏 — iPhone 的辅助功能
- **背景**：iPhone 需要支持无障碍用户，但不能让主界面变得复杂
- **策略**：辅助功能（VoiceOver、缩放、辅助触控）全部放在"设置→辅助功能"深处
- **结果**：主界面保持极简，需要的用户能通过"邀请探索"路径找到
- **关键点**：隐藏不等于删除——为特定用户保留入口，但不干扰主流用户

### 案例 4：转移 — Uber 的乘客-司机双端设计
- **背景**：打车涉及乘客和司机两个角色，复杂度不能放在一端
- **策略**：乘客端极简（选目的地、叫车、付款）；司机端承接导航、路线优化等重操作
- **结果**：乘客端上手零门槛，司机端功能完整
- **关键点**：转移的核心是"把复杂度放在能承受的角色/设备上"

### 案例 5：四策略组合 — Amazon 结账流程
- **删除**：结账页面去掉所有导航链接（减少跳出）
- **组织**：结账步骤按线性流程排列（地址→付款→确认）
- **隐藏**：优惠券输入框默认折叠，需要时展开
- **转移**：移动端扫描银行卡替代手动输入
- **结果**：结账完成率业界领先
- **关键点**：实际项目中通常需要四策略组合使用

### 案例 6：反面案例 — Microsoft Office 2003 → 2007 的 Ribbon
- **背景**：菜单层级太深，用户找不到功能
- **错误策略**：自动隐藏不常用菜单项（Office 2000 的尝试已失败）
- **正确策略**：用 Ribbon 重新组织功能（按行为分组而非按功能类型）
- **结果**：初期用户抱怨"找不到老功能"，但新用户上手更快
- **学习**：组织策略改变时需要给老用户过渡期；但不要因为老用户反对就停止优化

## 常见场景决策树（补充）

### 场景 A：用户说"找不到 XX 功能"
→ 诊断：
1. 功能是否存在？→ 不存在 → 考虑是否需要加
2. 功能存在但在哪？→ 设置深处 → 考虑"隐藏"策略的发现性问题
3. 功能存在且位置合理？→ 可能是标签/命名问题 → 用用户语言重新命名
→ 修复：在相关场景加"邀请探索"提示，或提升功能层级

### 场景 B：老板说"竞品有这个功能，我们也要加"
→ 应对：
1. 问：竞品的这个功能，用户实际使用率是多少？（通常 < 10%）
2. 问：加了这个功能，主流用户的体验会变好还是变复杂？
3. 问：能否用"隐藏"策略先放进去，观察使用率再决定是否提升？
→ 如果使用率 < 5% → 不加或深度隐藏

### 场景 C：产品功能越来越多，新用户上手困难
→ 诊断：
1. 新用户首次任务是什么？→ 这条路径是否足够突出？
2. 是否有"渐进展示"？→ 新用户看到的是核心功能还是全部功能？
3. 是否有"新手引导"？→ 引导是否 < 3 步？
→ 修复：组织+隐藏组合——核心功能突出，进阶功能渐进展示

### 场景 D：跨设备体验不一致
→ 诊断：
1. 手机端是否承担了"重操作"？→ 转移到桌面端
2. 状态是否实时同步？→ 技术问题而非设计问题
3. 用户是否知道"这个操作在哪个设备上做更好"？→ 加设备推荐提示

## 补充失败模式

| 失败信号 | 根因 | 修复 |
|---------|------|------|
| 删除后用户量下降 | 删掉了用户的"习惯依赖"功能 | 先做使用频率+用户分群分析，区分"偶尔用但需要"和"真没用" |
| 组织后用户仍迷路 | 分组逻辑正确但标签不直觉 | 做卡片分类测试，让用户自己分组 |
| 隐藏后用户投诉"功能没了" | 隐藏太深，缺乏发现线索 | 在相关场景加微交互提示（小图标、气泡） |
| 转移后设备间体验断裂 | 状态同步延迟或冲突 | 确保云端同步实时性，加冲突提示 |
| 设计评审通过但上线翻车 | 评审是内部视角，没做用户测试 | 上线前做 5 人可用性测试（成本低、收益高） |
| 简化后"看起来简单"但任务完成率低 | 简化了界面没简化流程 | 用任务完成率而非"界面复杂度"衡量简化效果 |
| 默认值选错导致用户出错 | 默认值基于开发便利而非用户场景 | 用最常见场景作为默认值，提供"更改"入口 |
| 新用户上手快但老用户流失 | 过度简化丢失了进阶功能 | 用渐进展示满足两类用户 |

## 补充检查点

### 删除策略执行后
- [ ] 是否监控了被删功能的"用户投诉率"？（投诉率 > 5% 考虑回滚）
- [ ] 删除后核心任务的完成率是否提升？
- [ ] 是否同步清理了相关的视觉噪音（多余线条、颜色、文字）？

### 组织策略执行后
- [ ] 是否做了卡片分类测试验证分组合理性？
- [ ] 用户能否在 3 秒内找到最常用的功能入口？
- [ ] 分组数量是否 ≤ 6？（超过则需要子分层）

### 隐藏策略执行后
- [ ] 隐藏功能是否有至少一种"发现线索"？（图标、动画、提示文字）
- [ ] 设置页是否仍在合理长度内？（超过一屏需要重新组织）
- [ ] 是否区分了"一次性设置"和"常用设置"？

### 转移策略执行后
- [ ] 各设备/角色的任务边界是否清晰？（用户不会困惑"该在哪做"）
- [ ] 状态同步是否实时？（延迟 > 5 秒用户会感知）
- [ ] 是否为每个设备提供了"最适合的任务"入口？
