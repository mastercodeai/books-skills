# 简约至上：交互式设计四策略

> **作者:** [英] Giles Colborne（科尔伯恩）
> **出版:** 人民邮电出版社，2013
> **原书:** Simple and Usable Web, Mobile, and Interaction Design

---

## 核心理念

简单不是最少化，而是在用户的头脑中创造一种**掌控感**。真正的简单发生在用户的世界里——让他们能毫不费力地完成目标，并留出空间让他们用自己的生活去填充体验。

- **顽固的复杂性**（Larry Tesler）：任何应用都有无法消除的复杂性，关键问题是**谁来面对这些复杂性**
- 简单 ≠ 最少化。简单的设计仍然可以有特征和个性（夏克椅 vs 潘顿椅）
- 简单 ≠ 貌似简单。说明书、操作向导、拟人角色都是"伪简单"

---

## 三种用户

| 类型 | 特征 | 占比 |
|------|------|------|
| **专家型** | 愿意探索，追求量身定做的技术，喜欢设定偏好 | 极少数 |
| **随意型** | 用过类似产品，对高级功能有兴趣但不愿接触全新事物 | 少数 |
| **主流型** | 为完成任务而用，只掌握核心功能，口头禅"能用就行" | **绝大多数** |

### 关键洞察

- **忽略专家型用户**：他们的判断有偏差，追求主流用户根本不在乎的功能（iPod发布时被专家嘲笑，最终卖出2.4亿台）
- **为主流用户而设计**：福特T型车——第一辆为平民大众制造的汽车，核心理念就是简单
- 用户类型不会随时间"升级"：用了5年Excel的人，大多数仍然只会对金额求和
- 简单的用户体验 = **初学者或压力之下的主流用户的体验**

### 主流用户 vs 专家

| 主流用户 | 专家 |
|----------|------|
| 立即把工作做完 | 首先设定偏好 |
| 容易操控最有价值 | 操控精确最重要 |
| 靠谱的结果 | 完美的结果 |
| 害怕弄坏什么 | 拆解一切的冲动 |
| 合适就行 | 精确匹配 |
| 看示例和故事 | 看原理 |

### 感情需求

主流用户最深层的需求是**掌控感**——掌控自己使用的技术 + 掌控自己的生活。

用"然后呢？"追问法挖掘感情需求：
> 任务太多 → 然后呢？ → 感觉失控 → 然后呢？ → 不想同时看到那么多 → 然后呢？ → 需要轻松的组织方式

## 四大策略总览

```
┌─────────────────────────────────────────────────┐
│   删除（Delete）    │  组织（Organize）          │
│  去掉不必要按钮      │  按标准将按钮分组          │
├─────────────────────────────────────────────────┤
│   隐藏（Hide）      │  转移（Displace）          │
│  非核心按钮藏仓盖下  │  基本按钮留遥控器，        │
│                     │  其余转移到电视菜单        │
└─────────────────────────────────────────────────┘
```

> 简化的秘诀：**删除不必要的、组织要提供的、隐藏非核心的、转移该转移的**

---

## 策略一：删除

**核心原则：聚焦于对用户有价值的功能**

Standish Group 研究：64%的软件功能"从未使用或极少使用"。

### 删除什么

1. **砍掉残缺功能**：不完整功能比没有更糟（沉没成本误区——"坏的可以修好，次品永远存在"）
2. **消除"假如用户想……"思维**：不要因想象中的场景保留功能，只保留目标用户**经常**遇到的问题的解决方案
3. **不要因"用户要求"就加功能**：对要求做逆向工程——搞清用户到底遇到了什么问题
4. **关注方案而非流程**：盯住流程会导致处理更多异常情况；退一步关注用户目的
5. **删除干扰因素**：多余的超链接、视觉混乱元素、不必要的文字
6. **砍掉选项和首选项**：选项过多 = 负担（果酱实验：24种→2%购买率，6种→12%购买率）
7. **删除错误来源**：设计不出错的界面 > 设计好的错误消息

### 删除文字的三大好处

- 重要内容"水落石出"
- 消除分析满屏内容的麻烦
- 读者对自己看到的更有自信

### 聪明的默认值

热门文档、类似项、个性化信息、共同选择、最近保存、恢复状态

### 避免删减过多

- 保留**控制感**：苹果无按钮电梯反面教材——删除操控装置≠简单
- 人们更愿意做**导航员**而非过路人

### 功能优先级排序法

1. 确定用户目的，排定优先次序
2. 专注寻找满足最高需求的方案
3. 确定最常见的干扰源，按难易排序
4. 区分"足够好"与"精准"的区别

## 策略二：组织

**核心原则：只强调一两个最重要的主题**

### 组织方法

| 方法 | 适用场景 | 注意 |
|------|----------|------|
| **分块** | 大量无序项 | 尽可能少分块（6组左右），越少越简单 |
| **围绕行为** | 理解用户想做什么、先做什么后做什么 | 理解用户心中的操作步骤 |
| **是非分明** | 分类标准 | 清晰界限，问用户怎么分类 |
| **时间** | 持续时间相近的活动 | 简单通用 |
| **空间** | 实体对象（酒店、国家） | 用户熟悉的空间布局 |
| **网格** | 布局对齐 | 吸引注意力，不对称避免局促 |
| **感知分层** | 复杂信息 | 用颜色/灰度/大小区分层次 |

### 大小与位置

- 重要元素大一些（即使比例失调）
- 不重要元素小一些（重要性1/2 → 大小1/4）
- 相似元素放在一起
- 导航位置：关注用户能否轻松找到按钮，而非固定左右

### 字母表与格式的陷阱

- 字母表排序：不知道想找什么的叫什么名就完蛋了
- 按格式（文字/图片/视频）排序：打断了信息的自然关联
- **期望路径**：观察用户实际怎么使用，而非设计者规划的路径

### 搜索 vs 浏览

- 没人始终把搜索作为第一选择
- 只有在导航无效时才用搜索
- **先组织好内容，再考虑搜索设计**

## 策略三：隐藏

**核心原则：彻底隐藏，适时出现**

### 适合隐藏的功能

- 不常用但不能少的功能（设置、偏好、地区信息）
- 一次性设计和选项
- 精确控制选项（但专家必须能始终看到）

### 不要做的事

- **不要让用户自定义界面**：主流用户更想展示个性（换壁纸），而非重排UI
- **不要自动定制**：Office 2000自适应菜单的失败——缩短菜单后用户要看两遍，位置会变

### 正确的隐藏方式

**渐进展示**：核心功能 + 扩展功能
- 保存对话框：先显示主流选项，专家点击展开
- 鼠标左键（基本）vs 右键（更多选项）
- Google高级搜索：主界面只有关键词搜索

**阶段展示**：随用户深入而逐步展示
- 设定场景、讲故事、说用户语言、分小块展示
- 每一步都符合用户心理预期

**适时出现**：只在合适的时机、合适的位置显示功能
- 《纽约时报》选词查词典：选中后出现问号图标
- 《纽约时报》的智慧：彻底隐藏 + 在你需要时恰好出现

### 让功能容易找到

- **标签放在哪里比做多大更重要**（用户关注点理论）
- 大标签如果不在用户关注点上 = 看不见
- 小标签如果在用户关注点上 = 效果好
- 用"邀请探索"而非"高级"标签

## 策略四：转移

**核心原则：把正确的功能放到正确的平台**

### 转移策略

1. **设备间转移**：RunKeeper——手机负责采集，网站负责展示分析
2. **向用户转移**：旅行规划程序——计算机管存储和检索，人管规划和估算

### 人 vs 计算机的分工

| 人擅长 | 计算机擅长 |
|--------|-----------|
| 设定目标和制定规划 | 执行程序 |
| 估算 | 精确计算 |
| 辨别信息 | 存储和检索信息 |
| 做图表 | 复制 |
| 在少数项中选择 | 大型列表排序 |
| 做预算 | 度量 |
| 想象 | 交叉引用详细信息 |

### 开放式体验

- **菜刀与钢琴**：专家和主流用户用同一工具设定不同目标，各得其乐
- 一个组件多种用途 > 多个单用途组件
- **非结构化数据**：让用户用自然方式输入，让计算机负责结构化
- 尽量减少仅适合中级用户的"便捷"特性

### 信任用户

- 让用户参与测试原型
- 让用户专注于选择和指挥，计算机专注于存储和计算
- 不要把用户当孩子

## 极端的可用性目标

| 常规目标 | 简单性目标 |
|----------|-----------|
| 特殊人群可以使用 | **任何人都可以使用** |
| 容易使用 | **毫不费力地使用** |
| 快速响应 | **瞬间响应** |
| 快速理解 | **一目了然** |
| 工作可靠 | **始终工作** |
| 直观的错误消息 | **不出错** |
| 完整的信息 | **恰好够用的信息** |
| 用户测试时工作 | **在混乱无序的环境中工作** |

> 瞄准极端目标——即使无法完全实现——也能帮你保持产品简单

## 明确认识的方法

1. **一句话描述**：简洁、清晰、完整（Flip = "拍摄和分享视频"）
2. **讲故事**：简明、具体、可信、有相关细节
3. **环境 + 角色 + 情节**：可信的环境、可信的角色、流畅的情节
4. **走出办公室**：在真实环境中观察用户
5. **分享认识**：反复讲述，让核心理念随处可见

## 设计哲学

1. **不要匆忙设计**：前3个方案通常只是对问题的初步描述，理解核心问题需要时间
2. **简单发生在用户的头脑中**：不要让你的设计干扰用户的思绪，留出空间让用户用自己的生活去填充
3. **细节决定成败**：微小的瑕疵会变成永远挥之不去的烦恼
4. **顽固的复杂性**：不是"怎么设计得更简单"，而是"复杂性应该放到哪里"
