# SKILL: 真需求分析（梁宁产品方法论）

## Description
运用梁宁《真需求》产品方法论进行需求识别、用户价值分析和产品决策。适用于产品规划、需求评审、用户研究等场景。

## Trigger
当用户需要：评估需求真伪、分析用户痛点、进行产品决策、设计用户价值主张、做PMF验证时触发。

## Scene Routing

根据用户问题自动路由到对应分析流程：

| 用户场景 | 路由目标 | 跳转 |
|----------|----------|------|
| "这个需求值不值得做" / 需求评审 | 需求真伪诊断 | → Steps 1-3 |
| "用户到底要什么" / 用户研究 | 痛点挖掘流程 | → Steps 1-2 + 痛点挖掘补充 |
| "产品方向对不对" / PMF验证 | PMF验证设计 | → Step 4 |
| "功能优先级怎么排" / 产品决策 | 决策输出 | → Steps 3 + 5 |
| "怎么设计用户价值" / 价值主张 | 价值分层分析 | → Step 2 + 决策输出 |
| 其他/模糊输入 | 先用四问法定位问题类型 | → Step 1 |

## Instructions

### Step 1: 需求诊断（四问法）
用"四问法"快速判断需求真伪：
1. **谁的痛？** → 定义具体用户画像，拒绝"所有人"
2. **有多痛？** → 1-10分评估痛苦程度，≤5分考虑是否为痒点
3. **现在怎么解决？** → 列出替代方案和迁移成本
4. **为什么是你？** → 明确差异化优势

### Step 2: 用户价值分层
将需求映射到三层价值：
- **功能价值**：解决什么具体问题？（必须回答）
- **情感价值**：用户用时什么感受？（体验设计空间）
- **社交价值**：用户会分享吗？（传播增长杠杆）

### Step 3: 痛点-痒点-爽点分类
| 类型 | 特征 | 产品策略 |
|------|------|----------|
| 痛点 | 不解决有损失，必须做 | 优先级最高，核心功能 |
| 痒点 | 解决了更好，可选 | 排期靠后，体验优化 |
| 爽点 | 超出预期，惊喜感 | 差异化设计，传播素材 |

### Step 4: PMF验证设计
- 定义北极星指标（1个核心指标）
- 设计"啊哈时刻"路径
- 先验证留存，再追求增长
- PMF信号：NPS > 40、自然推荐率 > 20%、周活跃占比 > 25%

### Step 5: 决策输出
输出结构化结论：
- 需求真伪判断（真/伪/待验证）
- 优先级建议（P0/P1/P2/不做）
- 核心价值主张（一句话）
- 最小验证方案

## 痛点挖掘补充（场景：用户研究）

当路由到痛点挖掘时，额外执行：

1. **行为观察优先于语言表达**
   - 看用户实际操作，不只听用户描述
   - 记录"卡顿点"：用户在哪里停下来、叹气、皱眉、切换到其他工具

2. **替代方案分析**
   - 用户现在用什么凑合？（Excel/微信群/手动记录/不处理）
   - 替代方案越粗糙，说明痛点越真实
   - 如果用户已有"还行"的替代方案，你面对的可能只是痒点

3. **频率 × 强度矩阵**
   | | 低频率 | 高频率 |
   |---|--------|--------|
   | 高强度 | 低频刚需（如报税），做好就行 | 高频强痛（金矿），P0 |
   | 低强度 | 忽略 | 高频微痛，用体验优化解决 |

4. **五层深挖法**
   - 用户说"报表不好用" → 问"哪里不好用？" → "数据不准" → "哪些数据不准？" → "上周的成交额对不上" → 根因可能是数据同步延迟，不是报表本身

## Failure Modes（失败模式）

| 失败模式 | 症状 | 诊断方法 | 解法 |
|----------|------|----------|------|
| **伪需求陷阱** | 做出来没人用 | 四问法第2题≤5分 | 降级为痒点或砍掉 |
| **用户说啥做啥** | 功能堆砌，没有核心主张 | 检查是否有1句话价值主张 | 回到"用户要解决的问题是什么" |
| **自嗨型产品** | 团队很兴奋，用户无感 | 没有PMF信号数据 | 做最小验证，看留存 |
| **痛点痒点不分** | 痛点痒点均匀投入资源 | 痛苦程度评分未做 | 重新评分，聚焦痛点 |
| **过早增长** | 拉新投入大但留存低 | 留存率 < 40% 还在投广告 | 停止拉新，修留存 |
| **解决方案锁定** | 需求文档直接写方案 | 没有"为什么"只有"做什么" | 回溯到用户问题层 |
| **幸存者偏差** | 只听活跃用户 | 没分析流失用户 | 做流失用户回访 |

## Anti-Patterns（反模式）

**❌ 直接问用户"你想要什么功能"**
→ ✅ 问"你上次遇到XX问题时怎么处理的？"

**❌ 用DAU衡量产品成功**
→ ✅ 用留存率和NPS衡量（用户是否真的需要你）

**❌ "先上线再说，迭代优化"**
→ ✅ 先验证核心假设是否成立，再决定是否投入

**❌ 把竞品功能列表当需求池**
→ ✅ 理解竞品功能背后的用户问题，再判断你的用户是否同样面临

**❌ 一个产品覆盖所有用户场景**
→ ✅ 一个核心场景做到极致，再扩展

**❌ 用功能数量衡量产品竞争力**
→ ✅ 用核心场景的完成质量衡量

**❌ "用户增长=产品成功"**
→ ✅ 留存+口碑=产品成功，增长只是结果

## Checkpoints（检查清单）

### 需求评审前
- [ ] 用户画像是否具体（不是"所有人"）
- [ ] 是否有行为数据或访谈支撑（不是拍脑袋）
- [ ] 痛苦程度评分是否≥6分
- [ ] 用户现有替代方案是否已列出
- [ ] 差异化优势是否清晰

### 产品决策时
- [ ] 核心价值主张能否用一句话说清
- [ ] 功能/情感/社交三层价值是否至少有一层强
- [ ] 是否有北极星指标
- [ ] 资源是否聚焦在痛点而非均匀分布

### PMF验证期
- [ ] 是否定义了"啊哈时刻"路径
- [ ] 留存率是否在跟踪
- [ ] NPS是否>40（未达到则聚焦修复）
- [ ] 是否在留存验证前就投入了拉新预算（如果是，暂停）

### 上线前
- [ ] 核心假设是否可被验证/证伪
- [ ] 最小验证方案是否足够"最小"
- [ ] 是否有明确的"继续/调整/放弃"决策标准

## Examples

**输入**：用户说想要一个"社交功能"
**分析**：
- 谁的痛？→ 现有用户在使用中感到孤独/缺乏反馈
- 有多痛？→ 如果是工具型产品，可能是痒点（3分）；如果是社区型产品，是痛点（8分）
- 现在怎么解决？→ 用户可能在微信群、朋友圈分享
- 结论：如果是工具产品，优先做好功能价值；如果是社区产品，社交是核心
- ⚠️ Anti-pattern check: 不要因为"竞品有社交"就做，先验证你的用户是否真的需要

**输入**：B端客户说"报表不好用"
**分析**：
- 观察行为：客户是导出Excel自己改，还是直接不用？
- 如果导出改 → 真痛点（7分），改报表功能是P0
- 如果不用 → 可能报表本身不是需求，背后的数据获取才是
- 五层深挖："不好用"→"数据不准"→"成交额对不上"→"上周的"→根因：数据同步延迟
- Checkpoint: 确认不是解决方案锁定，回到问题层再定义方案

**输入**：老板说"AI这么火，我们要加AI功能"
**诊断**：典型反模式——"把竞品功能列表当需求池"+"解决方案锁定"
**应对**：
1. 问：我们的用户面临什么AI能解决的问题？
2. 问：用户现在的替代方案是什么？痛苦程度几分？
3. 如果回答不了以上两个问题 → 暂不启动，先做用户调研

## Constraints
- **禁止拍脑袋定需求**：必须有用户行为数据或深度访谈支撑
- **警惕用户说的**：用户表达的是解决方案，不是需求本身
- **一个产品一个核心主张**：拒绝功能堆砌
- **留存优先**：留存没过验证线，不投入大规模拉新
- **痛苦程度是硬门槛**：≤5分的需求不进P0
- **验证先行**：核心假设未验证前，不投入全量开发资源

## 实战案例库

### 案例 1：微信红包 — 从痛点到爽点的跃迁
- **需求诊断**：谁的痛？→ 春节发红包的社交压力（传统红包金额攀比）
- **价值分层**：功能价值（便捷发红包）3分 → 社交价值（拼手气的乐趣）9分
- **关键洞察**：梁宁强调的"爽点"——超出预期的惊喜感（抢到随机金额的刺激）
- **PMF信号**：上线 2 天绑定 2 亿银行卡，自然传播率极高
- **学习**：社交价值 > 功能价值；"好玩"比"好用"传播力更强

### 案例 2：拼多多 — 痛点分级的极致实践
- **需求诊断**：谁的痛？→ 价格敏感用户（三四线城市、中老年群体）
- **痛苦程度**：9分——"同样的东西为什么比超市贵？"
- **替代方案**：线下批发市场、淘宝比价（迁移成本低但费时）
- **差异化**：拼团机制让"便宜"变成社交行为
- **学习**：真正的痛点用户往往不是"主流互联网用户"；被忽视的痛是金矿

### 案例 3：Notion — 痒点变痛点的时机把握
- **早期判断**：文档+数据库+看板整合 → 当时是痒点（5分），Word+Excel 够用
- **转折**：远程办公浪潮 → 痛点飙升到 8 分，"信息分散在 5 个工具"成为高频强痛
- **PMF信号**：自然推荐率 > 30%，用户自发创建模板社区
- **学习**：痒点在环境变化后可以变成痛点；PMF 有时是"等到了对的时机"

### 案例 4：一个 B 端反面案例 — "客户说要报表"
- **错误做法**：客户说"要报表"→ 产品团队花 3 个月做报表功能
- **五层深挖**："要报表"→"看数据"→"看趋势"→"判断该不该追加投放"→ 根因：客户真正需要的是"投放效果预警"
- **修正方案**：做自动预警通知，比报表有用 10 倍，开发量只有 1/5
- **学习**：用户说的是解决方案，不是需求；五层深挖是必做动作

### 案例 5：一个 SaaS 产品的 PMF 验证过程
- **北极星指标**：每周活跃用户数（WAU）
- **啊哈时刻**：用户在首次使用中完成"导入数据→看到第一个洞察"（<10 分钟）
- **实验 1**：注册后直接进仪表盘 → 激活率 15%（太低）
- **实验 2**：注册后先做"3 步引导" → 激活率提升到 45%
- **实验 3**：引导中加入"示例数据预填" → 激活率 62%，NPS 从 28 → 48
- **学习**：啊哈时刻需要精心设计路径，不是"功能有了用户自然会用"

## 常见场景决策树（补充）

### 场景 A：产品经理说"用户调研显示需求很强"
→ 追问：
1. 调研用的是态度数据（"你会用吗？"）还是行为数据（"你正在为这个付费吗？"）？
2. 受访者是否是真实目标用户？（朋友/同事 ≠ 目标用户）
3. 用户是否知道替代方案存在？（不知道替代方案的用户会高估需求）
→ 如果只有态度数据 → 设计一个行为验证实验

### 场景 B：老板/投资人说"这个市场规模很大"
→ 应对：
1. 大市场 ≠ 你的市场：TAM/SAM/SOM 是否清晰？
2. 你的切入点是大市场中的哪个痛点细分？
3. 竞争格局如何？已有玩家的弱点在哪？
→ 如果只有 TAM 数据 → 先验证 SOM（可获得市场）的需求真实性

### 场景 C：留存率 30%，不知道问题在哪
→ 诊断路径：
1. 30% 的留存用户有什么共同特征？（找到"超级用户"画像）
2. 流失用户在哪里流失？（注册后？首次使用后？一周后？）
3. 流失用户和留存用户的使用路径有何不同？
→ 重点：找到"啊哈时刻"——留存用户都经历过的那个关键时刻

### 场景 D：多个需求都是 P0，资源不够
→ 排序方法：
1. 按痛苦程度评分排序（必须 ≥ 7 分）
2. 按影响用户数量排序
3. 按"不做的后果"排序（用户流失 vs 用户不满 vs 无所谓）
4. 如果仍无法排序 → 做 A/B 测试或预售验证

## 补充失败模式

| 失败模式 | 症状 | 诊断方法 | 解法 |
|----------|------|----------|------|
| **数据幻觉** | 数据好看但用户无感 | 检查数据是否反映真实价值（注册 ≠ 使用） | 定义"有意义的行为"作为指标 |
| **需求镀金** | 一个简单需求做成大项目 | 检查需求文档的复杂度 vs 用户痛苦程度 | 用最小方案验证，再决定是否扩展 |
| **竞品焦虑** | 竞品做了我们也要做 | 问"竞品验证过这个需求对我们的用户也成立吗？" | 回到四问法独立判断 |
| **需求分层错误** | 把痒点当 P0 做 | 检查痛苦程度评分是否有行为数据支撑 | 用"替代方案分析"重新定级 |
| **PMF 幻觉** | 增长数据好看但靠补贴 | 停掉补贴看自然留存和推荐率 | 真PMF = 停补贴后留存仍 > 40% |
| **用户画像模糊** | "我们的用户是 25-45 岁白领" | 检查画像是否包含行为特征和场景 | 用"一天时间线"描述用户典型日 |
| **功能上线即遗忘** | 上线后没人跟踪使用率 | 检查是否有功能级别的使用率埋点 | 每个功能上线必须有"成功指标" |
| **需求来源单一** | 所有需求来自老板/销售 | 检查是否有用户行为数据驱动的需求 | 建立多渠道需求来源（数据+访谈+客服） |

## 补充检查点

### 四问法执行后
- [ ] 每个问题的回答是否有具体数据/案例支撑？（不是"感觉"）
- [ ] 替代方案是否包含"不处理"这个选项？（用户可能根本不在乎）
- [ ] 差异化优势是否是用户在意的？（不是团队自认为的）

### 价值分层后
- [ ] 三层价值中至少一层是否能用一句话向用户说清？
- [ ] 社交价值是否能自然触发分享行为？（不是靠"分享给好友得红包"）
- [ ] 情感价值是否有对应的体验设计空间？（不只是功能交付）

### PMF 验证中
- [ ] 啊哈时刻是否有明确的用户路径设计？
- [ ] 是否在跟踪"有意义的行为"而非虚荣指标？
- [ ] 留存曲线是否在同期群维度上趋于平稳？（而不是累计数据假象）
- [ ] 是否对流失用户做了至少 10 次深度访谈？

### 需求优先级决策后
- [ ] 被排在 P0 的需求是否都通过了痛苦程度 ≥ 7 的门槛？
- [ ] 是否有"不做"的明确理由记录？（防止需求反复出现）
- [ ] 资源分配是否真的聚焦在 P0？（检查团队实际工作分布）
