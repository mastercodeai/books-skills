# SKILL.md — 产品管理方法论（Marty Cagan）

> 来源：《启示录：打造用户喜爱的产品》Marty Cagan
> 核心能力：产品发现、产品交付、赋能团队、产品思维、原型测试

---

## Skill: Product Discovery & Delivery Framework

### Description
Marty Cagan 的产品管理方法论，源自硅谷顶级科技公司（eBay、网景、惠普）的实战经验。强调通过系统化的产品探索流程，打造兼具价值性、可用性、可行性的产品。适用于软件产品团队（互联网服务、企业级软件、消费电子产品等）。

### Scene Routing

| 场景 | 路径 | 核心动作 | 关键产出 |
|------|------|----------|----------|
| **产品发现** (Product Discovery) | 新方向/新功能探索 | 机会评估 → 原型 → 用户测试 | 验证过的原型 + 机会评估文档 |
| **原型测试** (Prototype Testing) | 验证假设、收集用户反馈 | 高保真原型 → 用户访谈 → 迭代 | 用户测试报告、通过/放弃决策 |
| **团队赋能** (Team Empowerment) | 团队角色混乱、效率低下 | 角色厘清 → 流程建立 → 文化塑造 | 团队宪章、产品原则、RACI矩阵 |
| **产品交付** (Product Delivery) | 确认可行后进入开发 | 基本产品定义 → 敏捷交付 → 平滑部署 | 可交付产品、发布计划、反馈闭环 |
| **产品转型** (Product Transformation) | 从功能工厂转向产品驱动 | 诊断现状 → 试点改变 → 全面推广 | 转型路线图、试点成果报告 |

**路由规则：**
- 不确定要做什么？→ 先走 **产品发现**
- 有方案但不确定行不行？→ 走 **原型测试**
- 团队不知道怎么配合？→ 走 **团队赋能**
- 已确认要做什么？→ 进入 **产品交付**
- 组织层面想改变？→ 走 **产品转型**

### When to Use
- 定义新产品或新功能前
- 产品团队角色不清、职责混乱时
- 产品频繁失败，无法达到目标时
- 需要建立系统化的产品开发流程时
- 创业公司产品方向不明确时
- 大公司创新受阻时
- 团队需要从"功能工厂"转向"产品驱动"模式时

### Core Principles

1. **产品探索先于产品执行**
   - 分两阶段：探索正确的产品 → 正确地开发产品
   - 探索阶段不可设定精确期限——它是创造性工作
   - 用高保真原型替代纸质文档作为产品定义核心

2. **价值、可用、可行三重验证**
   - 价值（Valuable）：用户是否渴望使用
   - 可用（Usable）：用户能否明白如何使用
   - 可行（Feasible）：技术上能否实现
   - 三者缺一不可，确认前不得进入开发

3. **赋能产品团队**
   - 产品经理与开发团队平等——定义正确的产品 vs 正确地开发产品
   - 交互设计不能外包——需深度理解用户
   - 预留20%开发资源用于技术债务
   - 让开发人员直接面对用户

4. **用户驱动而非技术驱动**
   - 产品经理必须亲自与用户交流
   - 使用人物角色（Personas）理解典型用户
   - 特约用户（6-10人）贯穿产品全流程
   - 观察用户操作，少听用户抱怨——用户不知道自己要什么

5. **情感是购买驱动力**
   - 企业消费者：恐惧和贪婪
   - 大众消费者：社交、爱情、自豪、获利等多样化情感
   - 关注非理性消费者——他们放大的情感需求代表大众潜在需求
   - 创新最佳土壤：失望、不满、愤怒等负面情绪

### Key Frameworks

#### 1. 机会评估（Opportunity Assessment）
十个核心问题，只讨论问题不涉及方案：
1. 产品要解决什么问题？（价值）
2. 为谁解决？（目标市场）
3. 成功机会多大？（市场规模）
4. 如何衡量成功？（指标）
5. 有哪些同类产品？（竞争）
6. 为什么我们最适合？
7. 时机合适吗？
8. 如何推向市场？
9. 成功的必要条件？
10. 继续还是放弃？

#### 2. 产品原则（Product Principles）
- 团队信仰和价值观的总结
- 不是功能清单，是战略指南
- 必须按重要性排序
- 用于统一团队决策标准

#### 3. 产品评审团（Product Council）
- 各部门管理者组成（≤10人）
- 四个里程碑评审：机会评估 → 方案定义 → 原型评审 → 发布评审
- 只做宏观决策，不干预具体设计

#### 4. 原型测试流程
1. 物色多样化真实目标用户
2. 少说话，不透露线索
3. 观察操作，不提示不引导（鹦鹉技巧：复述用户行为）
4. 从肢体语言和语气判断反应
5. 快速响应反馈，连续6位满意即通过
6. 无法让用户产生兴趣→立即放弃

#### 5. 基本产品（Minimal Product）
- 只满足基本要求的不可分割整体
- 如果还能削减，说明不是基本产品
- 定义完成后不允许随意修改

#### 6. 情感接纳曲线
- 技术爱好者→非理性消费者→理性消费者→超理性消费者→观望者
- 非理性消费者最值得研究——他们的情感需求驱动产品跨越鸿沟
- 技术爱好者会误导——痴迷技术而非解决问题

### Failure Modes & Recovery

| 失败模式 | 信号 | 根因 | 恢复策略 |
|----------|------|------|----------|
| **功能工厂模式** | 需求从上而下、团队被动执行 | PM沦为需求传话筒 | 从机会评估重建流程，争取一次"说不"的机会 |
| **原型不测试就开发** | 担心延误进度，跳过用户验证 | 对探索阶段价值不信任 | 先用一次"原型+测试"证明省下的返工成本 |
| **技术驱动产品** | 开发完才发现没人要 | 技术团队主导产品方向 | 让开发者直接观察用户，重建"三重验证"机制 |
| **设计外包依赖** | UX设计与产品理解脱节 | 认为设计是"美工活" | 引入内部交互设计师，参与用户研究全流程 |
| **需求反复变更** | 开发中不断改需求 | PM未充分探索就承诺 | 严格遵循"探索→确认→开发"节奏 |
| **发布即结束** | 发布后资源立刻撤走 | 缺乏发布后迭代意识 | 留1周全员收集反馈，建立持续改进机制 |
| **照单全收客户要求** | 定制功能越来越多 | 把客户当用户、把销售当PM | 产品经理必须直接与终端用户交流 |
| **忽视技术债务** | 速度越来越慢、bug越来越多 | 只追功能不还债 | 强制预留20%资源用于技术债务 |
| **PRD驱动开发** | 文档厚但没人看 | 用文档替代原型 | 用高保真原型替代PRD，可测试、无歧义 |
| **角色混搭** | 一人兼PM和营销 | 资源不足想省人 | 两个角色必须分开，否则哪个都做不好 |

### Anti-Patterns to Avoid

1. **市场营销定义产品** — 缺少详细需求和UX设计，产品浮于表面
2. **两人分担产品定义** — 无人对最终产品负责，决策效率低
3. **一人兼任PM和营销** — 扩展性差，精力不足，两个角色都做不好
4. **跳过机会评估直接开发** — 方向错误的代价远大于评估的时间
5. **瀑布式探索产品** — 验证严重滞后，问题发现太晚
6. **开发中不断修改需求** — PM应先做好原型，确认后再开发
7. **把开发初期产品当原型** — 时间太长、不可丢弃、修改成本高
8. **根据客户要求照单全收** — 用户不知道自己要什么，客户≠用户
9. **忽视技术债务** — 代码重写代价惨重，预留20%资源还债
10. **发布后立即撤走资源** — 错失改进最佳时机
11. **让技术爱好者主导需求** — 他们痴迷技术而非解决问题
12. **用Word/PRD定义产品** — 歧义多、不可测试、变更成本高

### Checkpoints

#### 阶段一：产品发现（Discovery）
- [ ] 机会评估十个问题全部回答清晰
- [ ] 目标用户有具体的人物角色定义
- [ ] 产品原则已确立并按重要性排序
- [ ] 高保真原型已构建（非纸质草图）
- [ ] 原型已通过至少6位目标用户测试
- [ ] 价值、可用、可行三重验证全部通过
- [ ] 技术可行性已与开发团队确认

#### 阶段二：方案定义（Definition）
- [ ] 基本产品（Minimal Product）定义明确
- [ ] 产品评审团已审核并通过方案
- [ ] 指标体系已确立（如何衡量成功）
- [ ] 特约用户（6-10人）已确定
- [ ] 开发团队理解"为什么做"而不只是"做什么"

#### 阶段三：产品交付（Delivery）
- [ ] UX设计在开发之前完成（第零次迭代）
- [ ] 开发人员参与了用户测试环节
- [ ] 20%开发资源已预留用于技术债务
- [ ] 平滑部署方案已制定（并行版本、增量部署、提前通知）
- [ ] 发布后1周反馈收集计划已安排

#### 阶段四：发布后迭代（Post-Launch）
- [ ] 用户反馈已收集并分析
- [ ] 关键指标数据已跟踪
- [ ] 改进优先级已基于数据（而非直觉）确定
- [ ] 团队已复盘探索和交付过程

### Decision Checklist

当面临产品决策时，依次检查：
- [ ] 是否明确了产品要解决的问题？（不只是功能描述）
- [ ] 是否明确了目标用户？（具体的人物角色）
- [ ] 是否做了机会评估？（十个问题）
- [ ] 是否创建了高保真原型？
- [ ] 是否请真实用户测试了原型？
- [ ] 是否验证了技术可行性？
- [ ] 是否有清晰的产品原则？
- [ ] 团队对目标和优先级是否达成共识？

### Tactical Tips

- **产品经理与产品营销必须分开**：前者定义产品，后者推广产品
- **产品经理与项目管理必须分开**：前者探索产品，后者执行交付
- **使用高保真原型替代PRD**：可测试、无歧义、缩短上市时间
- **UX设计先于开发**：敏捷中用"第零次迭代"完成设计
- **平滑部署**：并行版本、增量部署、提前通知
- **快速响应**：产品发布后留1周全员收集反馈
- **改进产品从数据出发**：不一味加功能，提升关键指标
- **创业公司先探索再开发**：只有PM+设计师+原型开发者，确认后再招程序员
- **大公司创新**：20%法则、臭鼬工程、主动观察用户
- **拒绝特例产品**：为一个客户定制功能会偏离正轨
- **平台产品**：终端用户必须排第一，不是开发人员

### One-Line Summary
> 产品经理的任务是在最短时间内，通过高保真原型和用户测试，确认产品兼具价值性、可用性和可行性，然后交给赋能团队正确地开发出来。
