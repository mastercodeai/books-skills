# SKILL.md — 用户体验要素 (The Elements of UX)

## 核心能力

运用 Jesse James Garrett 的五层模型（Strategy → Scope → Structure → Skeleton → Surface）进行系统化的用户体验设计，确保每个决策都有明确的战略依据。

## 场景路由

根据当前任务类型，直接进入对应层面的工作流程：

| 场景 | 入口层面 | 核心动作 | 输出物 |
|------|---------|---------|--------|
| "新功能/产品要做UX设计" | 战略层 → 逐层下探 | 从目标和用户需求出发，逐层定义 | 完整设计文档链 |
| "这个功能要不要做/优先级怎么排" | 范围层 | 需求评估、优先级排序、规格定义 | 功能规格说明书 |
| "页面跳转逻辑混乱/用户迷路" | 结构层 | 信息架构审查、交互流程梳理 | 架构图 + 流程图 |
| "界面布局不合理/找不到按钮" | 框架层 | 界面元素重设计、导航体系优化 | 线框图 |
| "视觉不好看/风格不统一" | 表现层 | 视觉规范制定、设计语言统一 | 风格指南 |
| "现有产品体验不好，要诊断" | 快速诊断框架 | 从战略层逐层排查 | 诊断报告 + 改进路线图 |
| "需要说服团队/领导重视UX" | 战略层 | 用商业目标和用户数据建立论据 | UX商业论证 |

**路由规则：先确认问题出在哪一层，再从该层入手。低层优化无法修复高层错误。**

## 工作流程

### 阶段一：战略层——明确目标
1. 与决策者对齐产品目标（商业目标 + 品牌识别）
2. 定义成功标准（可量化的、与用户行为绑定的指标）
3. 用户研究 → 用户细分 → 创建人物角色（Personas）
4. 输出：战略文档/愿景文档

### 阶段二：范围层——定义边界
1. 将战略转化为具体的功能规格（功能型）或内容需求（信息型）
2. 用乐观、具体、可验证的语言撰写功能规格
3. 评估需求优先级：战略匹配 × 用户需求 × 可行性
4. 识别并排除"范围蠕变"风险
5. 输出：功能规格说明书 / 内容需求文档

### 阶段三：结构层——设计架构
1. 功能型：定义概念模型 → 交互流程 → 错误处理策略
2. 信息型：选择架构方法（从上到下 + 从下到上结合）→ 确定结构类型
3. 建立受控词典和元数据体系
4. 输出：架构图（Architecture Diagram）

### 阶段四：框架层——细化界面
1. 界面设计：选择正确的界面元素，优先处理常见场景
2. 导航设计：全局 + 局部 + 上下文导航系统
3. 信息设计：分组、排列、指示标识（Wayfinding）
4. 输出：线框图（Wireframe）

### 阶段五：表现层——感知设计
1. 视觉设计检查：用户视线首先落在哪？是否对齐战略重点？
2. 对比与一致性平衡：吸引注意 vs 保持统一
3. 配色方案 + 排版 + 栅格布局
4. 输出：设计合成品（Design Comp）+ 风格指南（Style Guide）

## 各层检查点（Checkpoints）

### 战略层检查点 ✅
- [ ] 商业目标和用户需求是否同时被明确表述？
- [ ] 是否基于真实用户研究（非假设）创建了人物角色？
- [ ] 成功标准是否可量化、可追踪、与用户行为绑定？
- [ ] 团队成员是否对战略目标达成共识？
- [ ] 品牌识别是否与用户期望一致？

### 范围层检查点 ✅
- [ ] 功能规格是否使用乐观、具体、可验证的语言？
- [ ] 每个需求是否可追溯到战略层的某个目标？
- [ ] 是否有明确的优先级排序框架？
- [ ] 范围边界是否清晰——"不做什么"是否已定义？
- [ ] 是否存在"镀金"（超出必要范围的过度设计）？

### 结构层检查点 ✅
- [ ] 概念模型是否与用户的心智模型一致？
- [ ] 主要用户任务的步骤数是否已最小化？
- [ ] 错误处理策略是否覆盖了关键失败场景？
- [ ] 信息架构是否经过卡片排序等方法验证？
- [ ] 受控词典是否已建立并被团队认可？

### 框架层检查点 ✅
- [ ] 最常用的功能是否在最显眼的位置？
- [ ] 导航是否让用户随时知道"我在哪、能去哪、怎么回去"？
- [ ] 界面元素是否使用了用户熟悉的模式（非自创控件）？
- [ ] 信息层级是否通过视觉手段清晰表达？
- [ ] 线框图是否覆盖了正常流程 + 异常流程？

### 表现层检查点 ✅
- [ ] 视觉层级是否引导用户首先看到最重要的内容？
- [ ] 配色方案是否考虑了色盲/无障碍访问？
- [ ] 排版是否在可读性和美感之间取得平衡？
- [ ] 设计是否在不同屏幕尺寸下保持一致性？
- [ ] 风格指南是否足够完整，能保证其他设计师的产出一致？

## 核心方法清单

### 用户研究
| 方法 | 适用阶段 | 深度 | 成本 |
|------|---------|------|------|
| 问卷调查 | 战略层 | 浅/广 | 低 |
| 焦点小组 | 战略层 | 中 | 中 |
| 现场调查 | 战略层 | 深 | 高 |
| 任务分析 | 战略/范围层 | 深 | 中 |
| 卡片排序 | 结构层 | 中 | 低 |
| 用户测试 | 任何层 | 中-深 | 中 |
| 可用性测试 | 框架/表现层 | 中 | 中 |

### 设计文档体系
| 文档 | 对应层面 | 核心内容 |
|------|---------|---------|
| 战略文档 | 战略层 | 产品目标 + 用户需求 + 成功标准 |
| 功能规格 | 范围层 | 功能描述（乐观/具体/可验证） |
| 内容需求 | 范围层 | 内容类型/规模/负责人/更新频率 |
| 架构图 | 结构层 | 概念结构与节点关系 |
| 线框图 | 框架层 | 界面+导航+信息设计的整合 |
| 设计合成品 | 表现层 | 最终视觉呈现 |
| 风格指南 | 表现层 | 全局视觉标准与规范 |

## 设计原则

1. **自下而上建设**：每一层在下一层结束前完成（非开始前）
2. **连锁效应意识**：每个决策对上下层都有影响
3. **一致性优先**：内部一致 > 外部一致 > 创新
4. **用户语言**：使用受控词典，避免内部术语
5. **概念模型**：保持交互模式的一致性，谨慎使用比喻
6. **对比引导**：用视觉对比引导用户注意力到关键元素
7. **错误预防**：设计不可能犯错 > 使错误难以发生 > 帮助恢复
8. **文档适度**：定义比文档厚度更重要，根据实际需要撰写

## 失败模式

每个层面都有典型的失败模式，识别它们有助于快速定位问题：

| 层面 | 失败模式 | 症状 | 根因 |
|------|---------|------|------|
| 战略层 | 目标漂移 | 产品功能越来越多但用户不增长 | 战略目标模糊，用功能数量代替用户价值 |
| 战略层 | 用户画像虚构 | 设计评审时无人质疑但上线后无人用 | 人物角色基于假设而非真实研究 |
| 范围层 | 范围蠕变 | 项目不断延期，需求持续膨胀 | 缺少优先级框架，无法说"不" |
| 范围层 | 镀金工程 | 交付精美但用户不关心的功能 | 设计团队追求完美而非用户价值 |
| 结构层 | 心智模型错配 | 用户反复在同一个地方迷路 | 设计者的心智模型 ≠ 用户的心智模型 |
| 结构层 | 架构债务 | 新功能越来越难加进去 | 早期架构未考虑扩展性 |
| 框架层 | 自创控件 | 用户不知道怎么操作某个元素 | 用创新代替可用性，打破用户预期 |
| 框架层 | 导航黑洞 | 用户进入某页面后不知道怎么回去 | 缺少全局导航和面包屑等定位手段 |
| 表现层 | 美观陷阱 | 设计很好看但转化率低 | 视觉设计未服务于用户任务，美感凌驾于功能 |
| 表现层 | 一致性崩塌 | 产品不同页面像不同产品 | 缺少风格指南或风格指南未被遵守 |

## 反模式（Anti-patterns）

### ❌ 层面跳跃
**表现**：跳过战略和范围，直接画高保真设计稿。
**为什么有害**：漂亮的界面掩盖了错误的方向，返工成本极高。
**正确做法**：每层的输出是下一层的输入，不允许跳层。

### ❌ 自下而上的决策
**表现**：由技术实现难度决定功能取舍，由前端框架决定交互模式。
**为什么有害**：技术约束应该影响范围层的可行性评估，而非主导战略。
**正确做法**：先定义"用户需要什么"，再评估"技术怎么实现"。

### ❌ 领导审美驱动
**表现**：最终设计方案取决于最高薪那个人的审美偏好。
**为什么有害**：个人偏好 ≠ 用户偏好，且无法迭代验证。
**正确做法**：用用户研究数据和成功标准来评判设计，而非主观意见。

### ❌ 竞品抄袭式设计
**表现**：照搬竞品的设计方案，不考虑自身用户和战略差异。
**为什么有害**：竞品的UX可能是错的；即使对的，也可能不匹配你的用户。
**正确做法**：分析竞品的设计决策背后的逻辑，而非表面形态。

### ❌ 末端UX评估
**表现**：产品开发完成后才邀请UX团队"过一下"。
**为什么有害**：此时修改成本最高，且只能在表现层做微调。
**正确做法**：UX参与从战略层开始，贯穿整个产品生命周期。

### ❌ 用户测试万能论
**表现**：认为做了用户测试就等于做了UX设计。
**为什么有害**：用户测试是验证手段，不是设计方法；它无法替代架构设计和交互设计。
**正确做法**：用户测试是五层模型中的一环，用于验证各层决策，而非替代设计过程。

### ❌ 文档崇拜
**表现**：花大量时间写厚厚的UX文档，但团队不读不用。
**为什么有害**：文档是手段不是目的，过度文档化消耗设计时间。
**正确做法**：文档服务于沟通，根据团队实际需要决定详略程度。

## 常见陷阱

- ❌ 由现状决定设计（技术/组织结构驱动UX）
- ❌ 由模仿决定设计（照搬竞品）
- ❌ 由领导决定设计（个人喜好代替用户需求）
- ❌ 在项目末尾才做"用户体验评估"
- ❌ 用用户测试替代完整的UX设计过程
- ❌ 只关注表现层（视觉），忽略底层要素
- ❌ 范围蠕变：不控制需求导致项目失控

## 快速诊断框架

当面对一个UX问题时，按顺序排查：

1. **是战略问题吗？** → 我们知道用户要什么吗？目标清晰吗？
2. **是范围问题吗？** → 功能/内容定义是否明确？有没有范围蠕变？
3. **是结构问题吗？** → 信息架构/交互逻辑是否合理？
4. **是框架问题吗？** → 界面元素/导航/信息设计是否到位？
5. **是表现问题吗？** → 视觉设计是否引导了正确的注意力？

**核心判断：高层错误会削弱低层的正确决策，低层错误会让高层决策失去意义。**
