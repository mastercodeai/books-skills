# 用户体验要素 — 以用户为中心的产品设计

> Jesse James Garrett | 原书第2版 | 核心框架：五层模型（5 Planes）

---

## 一、核心思想

用户体验（UX）不是产品"看起来怎样"，而是产品"如何与外界发生联系并发挥作用"。以用户为中心的设计（User-Centered Design）要求在开发的每一个步骤中，都把用户列入考虑范围。

**关键认知：**
- 产品越复杂，确定良好体验就越困难——每个新增特性都增加了失败的机会
- 用户体验是可持续的竞争优势，特性竞赛只能带来短暂优势
- 用户遇到技术困难时会责怪自己——这种心理是最有效的"赶走用户"的方式
- 转化率是衡量UX效果最直接的商业指标

---

## 二、五层模型（Five Planes）

从底层到顶层，每一层都是上一层的基础。决策有自下而上的"连锁效应"。

```
┌─────────────────────────────────────┐
│  表现层 (Surface)    — 感知设计      │  ← 最具体
├─────────────────────────────────────┤
│  框架层 (Skeleton)   — 界面/导航/信息 │
├─────────────────────────────────────┤
│  结构层 (Structure)  — 交互/信息架构  │
├─────────────────────────────────────┤
│  范围层 (Scope)      — 功能规格/内容  │
├─────────────────────────────────────┤
│  战略层 (Strategy)   — 产品目标/用户需求│ ← 最抽象
└─────────────────────────────────────┘
```

### 1. 战略层（Strategy）

**两个核心问题：**
- 我们要通过这个产品得到什么？→ 产品目标
- 用户要通过这个产品得到什么？→ 用户需求

**产品目标三要素：**
- **商业目标**：赚钱或省钱，但需具体化
- **品牌识别**：品牌形象不应是无意形成的，要有意设计
- **成功标准**：可追踪的指标，绑定到可通过设计影响的用户行为

**用户需求研究方法：**
| 方法 | 特点 |
|------|------|
| 问卷/焦点小组 | 收集普遍观点，适合广度 |
| 现场调查(Contextual Inquiry) | 深入理解日常情境中的行为 |
| 任务分析 | 分解用户完成任务的精确步骤 |
| 用户测试 | 请用户测试产品/原型 |
| 卡片排序 | 探索用户如何分类信息 |

**用户细分维度：** 人口统计、消费心态、技术熟悉度、内容专业度、社会/专业角色

**人物角色（Personas）：** 基于用户研究创建的虚构人物，帮助团队在每个决策中始终以用户为中心思考。

### 2. 范围层（Scope）

**定义范围的两个理由：**
1. 知道正在建设什么——让团队有共同目标
2. 知道不需要建设什么——防止"范围蠕变"

**功能型产品 → 功能规格说明书：**
- 描述系统"将要做什么"而非"不应该做什么"（乐观）
- 尽可能具体，避免歧义
- 避免主观语气，功能规格必须可验证
- 量化定义（如"支持1000并发"代替"高性能"）

**信息型产品 → 内容需求：**
- 不要混淆格式（FAQ）和目的（提供常见问题解答）
- 估算内容规模（文字数、图片大小、文件字节）
- 确定每项内容的负责人和更新频率
- 建立内容清单（content inventory）

**需求优先级三重评估：** 战略目标匹配度 × 用户需求满足度 × 技术/资源可行性

### 3. 结构层（Structure）

**功能型产品 → 交互设计：**
- 描述"可能的用户行为"，定义"系统如何响应"
- **概念模型**：用户对交互组件工作方式的理解。保持一致性比创新更重要
- **错误处理三级策略**：① 设计成不可能犯错 ② 使错误难以发生 ③ 帮助用户从错误中恢复

**信息型产品 → 信息架构：**
- 基本单位是**节点**（可对应任意信息片段）
- **结构类型**：
  - 层级结构（树状）—— 最常见
  - 矩阵结构 —— 沿多维度移动，不超过3维
  - 自然结构 —— 无固定模式，适合探索
  - 线性结构 —— 书/文章/教学
- **两种架构方法**：
  - 从上到下（战略驱动）→ 可能忽略细节
  - 从下到上（范围驱动）→ 可能过于贴合现有内容
  - **最佳实践：两者结合**

**语言与元数据：**
- **受控词典**：网站使用的标准语言，防止内部术语侵入
- **元数据**："关于信息的信息"，越详细，架构灵活性越高
- 元数据 + 类词词典 = 更智能的搜索

### 4. 框架层（Skeleton）

三个紧密关联的要素：

| 要素 | 关注点 | 适用产品类型 |
|------|--------|-------------|
| 界面设计 | 提供"做某事"的能力 | 功能型 |
| 导航设计 | 提供"去某处"的能力 | 信息型 |
| 信息设计 | 传达想法给用户 | 两者都需 |

**习惯与比喻：**
- 界面要与用户已有习惯保持一致
- 内部一致性比外部一致性更重要
- 比喻（metaphor）要谨慎使用——减少猜测而非增加

**导航设计三目标：**
1. 提供在网站间跳转的方法
2. 传达元素与内容之间的关系
3. 传达内容与当前页面的关系

**导航系统类型：**
- 全局导航（覆盖整个网站）
- 局部导航（到附近地点）
- 辅助导航（快捷途径）
- 上下文导航（嵌入内容中的链接）
- 友好导航（联系信息、法律声明）
- 远程导航（网站地图、索引表）

**线框图（Wireframe）：** 整合界面设计、导航设计、信息设计的统一文档，是视觉设计和开发实施的参考。

### 5. 表现层（Surface）

**核心原则：忠于眼睛**
- 用户视线首先落在哪里？是否是战略上重要的元素？
- 成功设计的两个特征：① 视线沿流畅路径移动 ② 提供有效选择的引导

**对比与一致性：**
- **对比**：吸引注意力的主要工具，引导用户到关键部分
- **一致性**：内部一致（产品内部）+ 外部一致（跨产品/跨媒体）
- 栅格布局保证一致性，但不要教条遵守

**配色与排版：**
- 色彩是最有效的品牌识别工具
- 亮色 → 前景元素；暗色 → 背景元素
- 正文用简单字体，导航标签可用个性字体
- 不要用相似但不完全一样的风格

**风格指南（Style Guide）：** 承载所有设计决策的权威性文档，确保分散团队的设计一致性。

## 三、双重性

产品可分为两类，五层模型在每一层的具体要素不同：

| 层面 | 功能型产品 | 信息型产品 |
|------|-----------|-----------|
| 战略 | 产品目标 + 用户需求 | 同左 |
| 范围 | 功能规格 | 内容需求 |
| 结构 | 交互设计 | 信息架构 |
| 框架 | 界面设计 + 信息设计 | 导航设计 + 信息设计 |
| 表现 | 感知设计 | 同左 |

大多数产品是两者的混合体。

## 四、实践原则

### 正确的工作方式
- **自下而上建设**：每一层的工作在下一层结束前完成（而非开始前）
- **层面间的工作相互影响**：高层决策可能需要重新审视低层
- **文档不是目的**：定义才是。根据需求撰写正确级别的文档

### 三种错误的设计方式
- **由现状决定**（Design by Default）：UX跟随技术或组织结构
- **由模仿决定**（Design by Mimicry）：照搬其他产品
- **由领导决定**（Design by Fiat）：个人喜好代替用户需求

### 马拉松 vs 短跑
- 产品开发是马拉松，不是短跑
- 需要持续监控表现，适时调整节奏
- 深思熟虑的设计决策短期费时，长期省时
- 底层要素（战略、范围、结构）最难被感知，但决定成败

### 提出正确的问题
- 每个决策都要问："你为什么要这么做？"
- 测试不能替代考虑周密的设计过程
- 用户无法自己阐明需求——要"比用户更准确地理解他们的需求"

## 五、关键概念速查

| 概念 | 含义 |
|------|------|
| 用户体验 | 产品如何与外界发生联系并发挥作用 |
| 以用户为中心的设计 | 开发每一步都把用户列入考虑 |
| 转化率 | 衡量UX效果的直接商业指标 |
| 人物角色 | 代表真实用户需求的虚构人物 |
| 范围蠕变 | 需求不断膨胀导致项目失控 |
| 概念模型 | 用户对交互组件工作方式的理解 |
| 受控词典 | 网站使用的标准语言 |
| 元数据 | 关于信息的信息 |
| 线框图 | 整合框架层三要素的文档 |
| 风格指南 | 承载设计决策的权威文档 |
