# SKILL.md — 增长黑客方法论技能库

> 来源：《硅谷增长黑客实战笔记》曲卉

---

## 场景路由

遇到增长问题时，先定位场景，再跳转对应技能：

```
增长方向不明 / 团队目标分散 → 技能1（北极星指标）
需要量化分析增长机会 → 技能2（增长模型）
新用户流失严重 → 技能3（Aha时刻）→ 技能4（激活优化）
老用户流失 / 留存下降 → 技能5（留存分析）
获客成本高 / 想降低CAC → 技能6（推荐计划）+ 技能8（渠道优化）
需要验证假设 / 系统化实验 → 技能7（A/B测试）
决定投入哪些渠道 → 技能8（渠道选择）
要组建增长团队 → 技能9（团队组建）
```

**一句话判断**：如果还不知道增长该往哪走 → 先做技能1+2；如果知道方向但执行效果差 → 技能3~8按瓶颈选；如果组织能力不够 → 技能9。

## 技能1：定义北极星指标

### 适用场景
公司/产品增长方向不明确，团队各做各的，缺乏统一目标

### 6个评估标准
1. 反映产品核心价值（用户是否体验到价值？）
2. 反映活跃程度（非累积/静态指标）
3. 指标变好 = 公司整体向好
4. 易于理解和交流（优选绝对数）
5. 先导指标（非滞后指标）
6. 可操作

### 参考案例
| 产品 | 北极星指标 |
|------|-----------|
| Facebook | 月活跃用户数 |
| Airbnb | 订房天数 |
| Uber | 总乘车数 |
| Slack | 团队发送2000条消息 |
| Dropbox | 1台设备安装+1个文件 |

### 检验方法
将候选指标逐条对照6个标准打"是/否"，选择满足最多标准的指标

### 失败模式
- 选了收入/利润当北极星 → 滞后指标，无法驱动日常决策
- 选了注册用户数 → 虚荣指标，不反映活跃度
- 指标过于复杂 → 团队记不住，无法形成共识
- 多个指标并列 → 精力分散，失去聚焦意义

### 反模式
- ❌ 每个部门各自定义"北极星" → 其实没有北极星
- ❌ 领导拍脑袋定指标，不做6标准验证
- ❌ 指标定好后永不复盘，产品阶段变了指标没变

### 检查点
- [ ] 团队每个成员能脱口说出北极星指标
- [ ] 指标通过至少4/6条标准
- [ ] 有对应的子指标拆解到各团队
- [ ] 每季度复盘指标是否仍然适用

## 技能2：构建增长模型

### 适用场景
量化分析增长机会，决定聚焦领域，与管理层沟通增长策略

### 3步法
1. 定义北极星指标（输出变量）
2. 绘制用户旅程（从访客到核心价值的每一步）
3. 组装模型（为每步赋指标值，构建公式）

### 模板
```
[北极星指标] = 新增活跃用户 + 已有活跃用户
= (渠道流量 × 转化率₁ × 转化率₂ × ... × 转化率ₙ) + (已有用户数 × 留存率 × 活跃率)
```

### 使用技巧
- 目测找最大瓶颈变量
- 做What-if分析比较不同项目影响
- 按群组（渠道/平台/新老用户）拆解找洞察
- 每季度更新模型数据

### 失败模式
- 模型变量过多 → 复杂到没人用，沦为Excel摆设
- 数据不准确 → 错误输入导致错误决策
- 只建模不行动 → 分析瘫痪，永远在"看数据"
- 忽略变量间的交互效应 → 孤立优化某个环节反而伤害整体

### 反模式
- ❌ 建完模型就束之高阁，不用它指导实验优先级
- ❌ 用过时数据（半年前的转化率）做决策
- ❌ 只看平均值，不做群组拆解

### 检查点
- [ ] 模型包含用户旅程所有关键步骤
- [ ] 每个变量有真实数据支撑（非估算）
- [ ] 能用模型回答"做X能带来多少增长"
- [ ] 每月/季度更新数据，模型保持鲜活

## 技能3：找到Aha时刻和关键行为

### 适用场景
新用户流失严重，需要明确激活目标

### 4步法
1. **列出3~5个候选关键行为**（与核心价值相关）
2. **数据分析**：比较有/无该行为用户的留存曲线差异，差异最大的 = 关键行为
3. **定性调研**：问卷/访谈确认数据洞察背后的原因
4. **定义Aha时刻**：`(谁) 在 (多长时间) 内完成 (多少次) (什么行为)`

### 常见Aha时刻
| 产品 | Aha时刻 |
|------|---------|
| Facebook | 7天内加10个好友 |
| Twitter | 关注30个人 |
| Slack | 团队发送2000条消息 |
| Dropbox | 1台设备安装+1个文件 |
| LinkedIn | 1周内建立4个联系人 |
| Zynga | 下载次日返回 |

### 验证方法
设计实验推动更多用户完成关键行为 → 监测长期留存是否提升 → 验证因果性

### 失败模式
- 关键行为选错（相关性≠因果性） → 推动了错误行为，留存没改善
- Aha时刻门槛设太高 → 大部分用户达不到，指标无法改善
- 只看数据不做定性调研 → 知道"是什么"不知道"为什么"
- 定义后不做验证实验 → 可能是巧合而非因果

### 反模式
- ❌ 拍脑袋定Aha时刻，不做留存曲线对比
- ❌ 把"注册"当关键行为 → 注册不等于体验到价值
- ❌ 产品大改后不重新验证Aha时刻是否变化

### 检查点
- [ ] 关键行为有数据验证（留存差异显著）
- [ ] Aha时刻有具体数值（谁+时间+次数+行为）
- [ ] 设计过至少1次因果验证实验
- [ ] 团队对"什么行为代表用户获得价值"有共识

## 技能4：新用户激活优化（激动指数法）

### 适用场景
注册/激活漏斗转化率低

### 操作方法
1. 模拟新用户从广告到完成关键行为的全流程
2. 在每一步标注对激动指数的影响（+/-分值）
3. 累加总分：正=体验良好，负=需要优化
4. 重点优化扣分最多的环节

### 四大原则
1. **增强动力**：前后一致性、权限解释、社会信任、先给价值
2. **减少障碍**：推迟注册、合并步骤、避免冷启动、减少选择
3. **适时助推**：明确机会窗口、灵活UX模式、外部渠道提醒
4. **私人订制**：自选偏好、历史推荐、按目的分流

### 关键指标
- 激活率（首日/次日/首周完成关键行为的比例）
- 激活漏斗各步转化率
- 新用户留存指标（注册4周/60天后仍活跃）

### 失败模式
- 只优化注册漏斗，不管注册后体验 → 注册上去了但激活没变
- 过度减少步骤导致用户不了解产品 → "注册即流失"
- 推送/邮件太频繁 → 用户反感卸载
- 冷启动问题没解决 → 用户进来后不知道干什么

### 反模式
- ❌ 在激活流程中强推付费/升级 → 用户还没体验到价值
- ❌ 一刀切的引导流程，不分用户来源/目的
- ❌ 只关注漏斗转化率，不看激活后留存（短期优化伤害长期）

### 检查点
- [ ] 完整走一遍新用户流程，记录每步感受
- [ ] 激动指数总分为正
- [ ] 新用户能在首次session内完成关键行为（或知道怎么完成）
- [ ] 激活率提升后，留存率没有下降（防短期优化）

## 技能5：留存分析与改善

### 适用场景
用户流失快，需要系统性改善留存

### 留存曲线绘制（4步）
1. 记录每周/月首次完成关键行为的用户数
2. 追踪这些用户后续每期的活跃数量
3. 计算每期活跃占比
4. 画成曲线图

### 分析技巧
- **横向**：曲线是否趋于平缓？（向下→可能未达PMF）
- **纵向**：新群组是否比旧群组更平？（产品在改善）
- **分群**：按渠道/行为/特征对比留存曲线

### 提高留存8种武器（按优先级）
1. 产品改进（杠杆最大）
2. 新用户引导和教育
3. 邮件
4. 通知/推送（RRF：覆盖面×相关性×频率）
5. 客户服务
6. 促销
7. 忠诚客户计划
8. 新产品/功能

### 参与度阶梯迁移
```
消极用户（基础使用）→ 核心用户（正确使用）→ 超级用户（高频+多功能）
```
- 定义每层用户的行为特征
- 设计引导机制推动用户上移

### 失败模式
- 留存还没做好就开始拉新 → 漏桶效应，烧钱无底洞
- 只看总体留存曲线，不做分群 → 关键问题被平均值掩盖
- 过度依赖推送/邮件挽留 → 治标不治本，用户疲劳后更难救
- 把短期回访（7天）当留存 → 真正的留存要看30/60/90天

### 反模式
- ❌ 留存率达标就停止关注 → 竞争对手在改善，你不进则退
- ❌ 用补贴/促销拉留存 → 停止补贴后用户立即流失
- ❌ 只关注已流失用户的召回，不关注即将流失用户的预防

### 检查点
- [ ] 留存曲线趋于平缓（不会降到0）
- [ ] 新群组留存率 ≥ 旧群组（产品在变好）
- [ ] 明确知道留存的"关键行为"是什么
- [ ] 留存改善策略按优先级排序执行（不是撒胡椒面）

## 技能6：用户推荐计划设计

### 适用场景
降低获客成本，利用口碑效应

### 万能公式分解
```
推荐新用户 = 活跃用户数 × 接触邀请机会比例 × 邀请转化率 × 平均邀请数 × 接受率 × 注册转化率
```
→ 逐环节优化，找最大瓶颈

### 6种推荐类型
1. 口口相传（产品体验极好）
2. 展示相传（好奇心驱动）
3. 补贴推荐（功能奖励 > 现金奖励）
4. 社交网络推荐（邀请使双方体验更好）
5. 病毒传播（有趣+视觉冲击+易分享）
6. 产品内传播（产品需求、内容分享、欢乐时刻、顺便接触）

### 最佳实践
- 动态触发优于静态菜单（融入"可分享时刻"）
- 双向补贴效果好于单向
- 邀请流程每一步都要A/B测试

### 失败模式
- 病毒系数 K < 1 → 推荐不能自增长，只是额外获客渠道
- 补贴被薅羊毛 → 推荐用户质量极低，LTV为负
- 邀请流程太复杂 → 用户懒得分享
- 只关注拉新数量，不关注推荐用户的质量/留存

### 反模式
- ❌ 产品体验很差就推推荐计划 → 口碑是负的
- ❌ 用现金奖励而非产品内奖励 → 吸引薅羊毛用户
- ❌ 推荐计划上线后不迭代 → 初始设计几乎不可能是最优的

### 检查点
- [ ] 推荐用户的留存率 ≥ 自然获客用户
- [ ] 病毒系数 K ≥ 0.5（理想 ≥ 1）
- [ ] 推荐流程 ≤ 3步完成
- [ ] 有A/B测试计划持续优化每个环节

## 技能7：A/B测试与增长实验

### 适用场景
任何需要验证假设、优化指标的场景

### 增长冲刺流程
1. **产生想法**：全员参与，维护想法库（500+）
2. **优先级排序**：ICE评分（Impact影响力 × Confidence信心 × Ease容易度）
3. **设计开发**：最小可行性测试（MVP思维）
4. **分析数据**：关注统计显著性
5. **应用结果**：赢→全量发布，输→学到洞察→下一个实验

### 周节奏
- 周一：检查运行中实验，准备增长会议
- 周二：增长例会（看指标→回顾进展→分析结果→决定下周实验）
- 周三~五：设计开发下周实验
- 目标：每周上线≥3个实验

### 注意事项
- 60%以上实验会失败，这是正常的
- "没有万能灵药，唯有不断试验"
- 记录所有实验结果，形成团队知识库

### 失败模式
- 样本量不够就下结论 → 假阳性导致错误决策
- 同时测太多变量 → 无法归因哪个改变有效
- 只看短期指标不看长期影响 → 短期涨了但留存/满意度下降
- 实验想法枯竭 → 团队缺乏灵感来源，实验速度下降
- "赢得"实验没有全量发布 → 实验白做

### 反模式
- ❌ 看到p<0.05就宣布胜利，不考虑效应量和业务意义
- ❌ 实验失败后不复盘学习，直接下一个
- ❌ 只测UI/文案层面的小改动，不敢测产品方向级假设
- ❌ 没有统一的实验记录系统，知识散落在各处

### 检查点
- [ ] 每周上线 ≥ 3个实验
- [ ] 想法库持续增长（每月新增 > 消耗）
- [ ] 每个实验有明确假设、指标、最小样本量
- [ ] 赢/输实验都有文档记录和团队复盘
- [ ] 60天后回访"赢"实验的长期效果

## 技能8：选择和优化用户获取渠道

### 适用场景
确定获客渠道策略，优化CAC

### 五步法
1. 了解产品特点（对象/形态/行业/盈利/LTV）
2. 了解用户群体（画像/习惯/在哪）
3. 列出备选渠道（19种：付费/有机/其他）
4. 筛选初选渠道（大体量→免费→可追踪→精准→可随时启停）
5. 持续追踪CAC和LTV，优化+拓展

### 渠道筛选四象限
- 优先选择：低CAC + 高LTV（右上角）
- 持续追踪每个渠道的CAC和用户质量

### 四类可做大的渠道
1. 付费增长（Facebook/Google广告）
2. 病毒传播（社交/网络效应产品）
3. SEO（内容型产品）
4. 销售（高LTV企业产品）

### LTV/CAC健康比
- < 1：不可持续
- ≥ 3：健康的商业模式（风投期望值）

### 失败模式
- 渠道过度集中 → 依赖单一渠道，平台政策变化即崩盘
- 只看CAC不看LTV → 便宜的渠道来的用户质量差
- 渠道饱和后仍硬投 → 边际CAC飙升，ROI为负
- 忽略自然流量 → 过度依赖付费，一旦停止投放流量归零

### 反模式
- ❌ 盲目跟风竞品的渠道策略 → 产品/用户画像不同
- ❌ 只优化前端获客，不优化后端变现 → 永远跑不通LTV>CAC
- ❌ 对所有渠道等额预算分配 → 应该聚焦少数有效渠道

### 检查点
- [ ] 至少测试过5个以上渠道
- [ ] 每个渠道有CAC和LTV数据（按群组）
- [ ] 最大渠道占比 < 50%（防集中风险）
- [ ] LTV/CAC ≥ 3（或有明确路径达到）
- [ ] 渠道组合包含付费+有机两条腿

## 技能9：组建增长团队

### 前提条件检查
- [ ] 产品已达PMF（留存率达标）
- [ ] 领导层理解和支持
- [ ] 有可用的工程师/设计师资源
- [ ] 有明确的增长运作流程计划
- [ ] 有合适的增长负责人

### 基本配置
1 增长PM + 2~3 工程师 + 1 数据分析师 + 1 设计师

### 增长PM的核心素质
- 数据驱动（最重要）
- 有产品思维和商业敏感度
- 能身兼数职
- 有创业经验优先
- 对增长有热情和好奇心

### 三种组织架构选型
| 模式 | 适用 | 特点 |
|------|------|------|
| 独立特区 | 大型/早期就建立 | VP直报CEO，全栈团队 |
| 功能主导 | 中型，已有产品团队 | 归属产品部门，协调方便 |
| 内部咨询 | 初次尝试增长 | 咨询+布道，传播增长文化 |

### 关键成功因素
- 第一个月"打响第一炮"（快速胜利建立信任）
- 做好"宣传委员"（定期分享实验结果）
- 从"两不管"地带入手（避免与其他团队冲突）

### 失败模式
- PMF未达标就建增长团队 → 花钱加速一个没人要的产品
- 增长团队没有工程资源 → 变成PPT团队，无法快速实验
- 增长团队和其他团队对立 → 内部政治消耗精力
- 找不到合适负责人 → 领导力不足导致团队形同虚设
- 过早追求规模化 → 流程重于结果，实验速度下降

### 反模式
- ❌ 招了增长PM但不给决策权 → 推不动任何事
- ❌ 增长团队KPI与其他团队冲突 → 内耗
- ❌ 只关注实验数量不关注质量 → 为了实验而实验
- ❌ 短期看不到大成果就裁撤增长团队 → 增长是长期积累

### 检查点
- [ ] 产品留存曲线已平缓（PMF确认）
- [ ] 增长团队有独立的工程资源
- [ ] 有周度增长例会和实验节奏
- [ ] 第90天有可量化的增长成果（建立信任）
- [ ] 增长文化在组织中扩散（其他团队也在提增长想法）

## 跨技能检查清单

启动任何增长工作前，先过这5个问题：

1. **PMF确认了吗？** → 留存曲线平缓 = 产品有人要，没达到先别做增长
2. **北极星定了吗？** → 团队是否对齐在同一个指标上
3. **瓶颈找到了吗？** → 增长模型能指出最大的杠杆点
4. **数据就位了吗？** → 没有数据支撑的假设只是猜测
5. **资源到位了吗？** → 没有工程资源的实验计划是空想
