## 技能1：确定当前所处的精益分析阶段

### 适用场景
创业早期或增长遇到瓶颈时，判断下一步应该关注什么

### 六阶段检查清单

| 阶段 | 通过标准 | 下一步 |
|------|---------|--------|
| 移情（Empathy） | ≥40%用户回答"如果不能再用会非常失望"（Sean Ellis测试） | → 黏性 |
| 黏性（Stickiness） | 留存曲线趋于平缓，核心用户使用频率稳定 | → 病毒性 |
| 病毒性（Virality） | K-factor > 0.5 且病毒周期持续缩短 | → 收入 |
| 收入（Revenue） | LTV/CAC ≥ 3，CAC回收期 < 12个月 | → 规模化 |
| 规模化（Scale） | 单位经济效益在新市场/渠道依然成立 | → 持续优化 |

### 操作方法
1. 诚实回答："我们产品现在最弱的环节是什么？"
2. 用上面的检查清单确定当前阶段
3. **只关注该阶段的OMTT**，其他指标暂放
4. 达到通过标准后，才考虑进入下一阶段

### ❌ 失败模式（IF-THEN）
| IF 你观察到… | THEN 根因是… | 立即行动 |
|-------------|------------|---------|
| 留存曲线持续下降无平台期 | 未达到PMF就做增长 | 停止所有获客投放，回到移情阶段深挖用户需求 |
| K-factor高但留存差 | 产品是"虚荣性病毒"（用户尝鲜后离开） | 降低传播优先级，先修留存 |
| LTV/CAC < 1 | 每获取一个用户就亏钱 | 立即收缩获客渠道，只保留ROI最高的1-2个 |
| 同时追踪多个阶段指标 | 注意力分散 | 回到OMTT框架，砍到只剩1个核心指标 |
| 阶段判断依据靠"感觉" | 缺乏数据支撑 | 先部署Sean Ellis测试+留存曲线两个最小数据集 |

### 🚫 反模式黑名单
- ❌ **"增长黑客跳过黏性"**：在留存<30%时投广告拉新 = 往漏水桶里倒水
- ❌ **"阶段跳跃"**：没验证收入就上规模化 = 烧钱放大亏损
- ❌ **"多阶段并行"**：同时优化5个指标 = 什么都优化不了
- ❌ **"一次性诊断"**：只在项目启动时判断阶段 = 阶段会倒退，需持续检查

### ⏸ CHECKPOINT-1：阶段确认
> **通过标准**：能用一句话说清"我们在___阶段，OMTT是___，及格线是___"
> **自检**：团队其他人能给出相同答案吗？不能 → 回到技能1重新对齐

## 技能2：定义你的OMTT（一个关键指标）

### 适用场景
团队数据过载、指标太多反而不知道该看什么

### 操作步骤
1. **列出你当前阶段最关心的3~5个指标**
2. **筛选标准**：
   - 这个指标改善，业务是否整体向好？
   - 团队能否直接行动影响它？
   - 能否在一周内观察到变化？
   - 是否易于理解和沟通？
3. **选择得分最高的一个作为OMTT**
4. **为OMTT设定及格线**（基于行业基准或历史数据）
5. **在团队中公开透明地跟踪OMTT**

### 及格线参考值

| 指标 | 及格线 |
|------|--------|
| 留存率（月） | > 60%（SaaS），> 30%（消费级） |
| 免费转付费率 | 2~5% |
| 日活/月活比（DAU/MAU） | > 20% |
| 次日留存（移动应用） | > 25% |
| 转化率（电商） | 2~3% |
| NPS（净推荐值） | > 0（正值即可），> 50为优秀 |
| LTV/CAC | ≥ 3 |

### ❌ 失败模式（IF-THEN）
| IF 你观察到… | THEN 根因是… | 立即行动 |
|-------------|------------|---------|
| OMTT指标持续改善但业务没增长 | 选了虚荣指标当OMTT | 用技能3的3问重新验证OMTT选择 |
| 团队对OMTT理解不一致 | 指标定义模糊 | 写清计算公式+数据来源，全员对齐 |
| OMTT一个月没变化 | 指标太滞后或太宏观 | 拆解为更小的先导指标（leading indicator） |
| 频繁更换OMTT | 没有给实验足够时间 | 每次更换OMTT前至少跑2周实验 |

### ⏸ CHECKPOINT-2：OMTT锁定
> **通过标准**：OMTT已书面定义（含公式+数据源+及格线），全员知晓
> **自检**：新入职同事看一眼能理解吗？不能 → 简化定义

## 技能3：区分虚荣指标与可执行指标

### 适用场景
向管理层/投资人汇报数据时，或设计数据看板时

### 转化框架

| 虚荣指标 | → | 可执行指标 |
|---------|---|-----------|
| 总注册用户数 | → | 月活跃用户数、激活率 |
| 总页面浏览量 | → | 每用户页面浏览数、跳出率 |
| 总下载量 | → | 次日留存率、使用频率 |
| 总收入 | → | 每用户平均收入（ARPU）、毛利率 |
| 总粉丝数 | → | 互动率、分享率 |
| 总邮件列表数 | → | 打开率、点击率、退订率 |

### 判断标准
问自己三个问题：
1. **看到这个数字变化，我能做什么？**（如果不能 → 虚荣指标）
2. **这个数字是我自己能影响的吗？**（如果不能 → 虚荣指标）
3. **这个数字代表的是真实的价值创造吗？**（如果不确定 → 可能是虚荣指标）

### 🚫 反模式黑名单
- ❌ **"总数崇拜"**：只展示累计注册数/总融资额 = 掩盖了增长停滞
- ❌ **"平均数陷阱"**：看平均值忽略分布 = 少数大客户拉高均值，多数用户在流失
- ❌ **"环比自嗨"**：环比增长50%但基数只有10 = 增长的幻觉

## 技能4：设计和分析留存曲线

### 适用场景
判断产品是否达到PMF，指导产品改进优先级

### 操作步骤
1. **定义关键行为**：用户完成什么行为算"留存"（不是登录，而是核心使用行为）
2. **定义天然使用周期**：日频/周频/月频产品用不同的时间窗口
3. **绘制同期群留存曲线**：
   - X轴：时间（天/周/月）
   - Y轴：完成关键行为的用户占比
   - 每条线 = 一个同期群（同一时期注册的用户）
4. **解读曲线**：
   - 曲线趋于平缓 = 有一部分用户会长期留存（PMF信号）
   - 曲线一直下降 = 没有找到PMF
   - 新同期群曲线比旧的更平 = 产品在改善

### 留存诊断清单
- [ ] 留存率在第1周下降超过75%？→ 优化新用户引导
- [ ] 留存率在第2~4周持续下降？→ 产品核心价值未触达
- [ ] 留存率在第2~3个月才开始下降？→ 产品参与深度不够
- [ ] 留存率在不同渠道差异大？→ 渠道质量问题

### ❌ 失败模式（IF-THEN）
| IF 你观察到… | THEN 根因是… | 立即行动 |
|-------------|------------|---------|
| 留存曲线第1天就断崖下跌 | 新用户引导失败 | 分析新用户前3步行为，找到流失断点 |
| 留存曲线平缓但值很低（5%） | 产品只有极小众在用 | 考虑缩小目标市场或调整定位 |
| 不同同期群曲线差异巨大 | 产品/渠道/季节干扰 | 控制变量，按渠道分别画曲线 |
| "登录留存"好看但"行为留存"差 | 留存定义太宽松 | 重新定义留存 = 核心使用行为 |

### ⏸ CHECKPOINT-3：留存验证
> **通过标准**：留存曲线已绘制且能回答"曲线是否平缓？平缓在什么水平？"
> **自检**：如果曲线不平缓，能否说出3个具体的改进实验？不能 → 回到技能4深挖

## 技能5：计算和优化LTV/CAC

### 适用场景
验证商业模式可行性、决定获客预算、向投资人展示单位经济效益

### 计算公式

**SaaS模式**：
```
LTV = ARPU × (1 / 月流失率)
    = 每用户月收入 × 平均客户生命周期（月）
    
CAC = (营销总费用 + 销售总费用) / 同期新付费客户数

LTV/CAC 比率 = LTV / CAC

CAC回收期 = CAC / ARPU （几个月收回获客成本）
```

**电商模式**：
```
LTV = 平均客单价 × 年购买次数 × 平均客户年限 × 毛利率

CAC = 总获客费用 / 同期新客户数
```

### 健康标准
| 指标 | 不健康 | 及格 | 优秀 |
|------|--------|------|------|
| LTV/CAC | < 1 | 1~3 | > 3 |
| CAC回收期 | > 24个月 | 12~24个月 | < 12个月 |
| 毛利率 | < 40% | 40~70% | > 70% |

### 优化方向
- **LTV太低**：提高留存、提高客单价、降低流失率
- **CAC太高**：优化获客渠道、提高转化率、增加推荐/口碑
- **回收期太长**：缩短付费周期、提高前期转化

### ❌ 失败模式（IF-THEN）
| IF 你观察到… | THEN 根因是… | 立即行动 |
|-------------|------------|---------|
| LTV/CAC < 1 | 获客就是亏钱 | 立即暂停付费获客，转有机渠道 |
| LTV/CAC > 5 | 获客投入不足，增长太慢 | 适度加大获客预算，测试新渠道 |
| CAC回收期 > 18个月 | 现金流压力极大 | 缩短付费周期，提高首付/预付比例 |
| LTV计算用"平均值" | 被大客户拉高 | 分层计算LTV（高/中/低价值客户） |

### ⏸ CHECKPOINT-4：经济模型验证
> **通过标准**：LTV/CAC≥3 且 回收期<12月
> **未达标**：先优化分子（LTV）还是分母（CAC）？选影响更大的那个先动手

## 技能6：计算病毒系数（K-factor）并优化传播链

### 适用场景
评估产品的自发传播能力，设计增长飞轮

### 计算公式
```
K-factor = 邀请发送率 × 邀请接受率

其中：
- 邀请发送率 = 发送邀请的用户数 / 总用户数
- 邀请接受率 = 接受邀请并注册的人数 / 发送的邀请总数
```

**带病毒周期的复合增长**：
```
用户增长 = 初始用户 × (1 + K)^(时间/病毒周期)
```
病毒周期越短，增长越快。

### 传播链优化清单
- [ ] 用户在什么时刻最容易分享？（找到"可分享时刻"）
- [ ] 分享流程是否足够简单（1~2步完成）？
- [ ] 被邀请者的首次体验是否足够好？
- [ ] 是否有双向激励（邀请者和被邀请者都获得奖励）？
- [ ] 分享内容是否有吸引力（社交货币价值）？

### 各类产品的K-factor参考
| 产品类型 | 典型K-factor |
|---------|-------------|
| 社交网络 | 0.5~1.5 |
| 协作工具 | 0.3~0.8 |
| 电商 | 0.1~0.3 |
| 内容平台 | 0.2~0.5 |
| 企业SaaS | 0.1~0.3 |

### ❌ 失败模式（IF-THEN）
| IF 你观察到… | THEN 根因是… | 立即行动 |
|-------------|------------|---------|
| K高但留存差 | "薅羊毛病毒"——用户来拿奖励就走 | 降低激励力度，提高产品质量门槛 |
| 邀请发送率高但接受率低 | 被邀请者首次体验差 | 优化被邀请者的onboarding流程 |
| K在下降 | 分享疲劳或产品新鲜感消失 | 更新分享机制，引入新的分享触点 |

### 🚫 反模式黑名单
- ❌ **"买来的病毒性"**：靠高额红包拉人 = K-factor虚高，补贴停则增长停
- ❌ **"单向激励"**：只有邀请者得奖励 = 被邀请者动力不足，接受率低

## 技能7：构建商业模式数据仪表盘

### 适用场景
为团队建立数据看板、设计数据报告体系

### 仪表盘三层架构

**第一层：战略层（管理层看）**
- OMTT（一个关键指标）+ 及格线
- LTV/CAC 比率
- 月环比增长率

**第二层：战术层（产品/运营看）**
- 漏斗各步转化率
- 留存曲线
- 活跃用户分群

**第三层：执行层（工程师/设计师看）**
- A/B测试结果
- 功能使用频率
- 错误率和性能指标

### 数据看板设计原则
1. **一屏展示**：最重要的指标在第一屏
2. **趋势 > 快照**：看变化趋势，不要只看当前值
3. **对比 > 绝对值**：环比、同比、与基准对比
4. **红绿灯系统**：及格线上绿色，线下红色
5. **自动化**：日报/周报自动生成，不依赖人工

### ❌ 失败模式（IF-THEN）
| IF 你观察到… | THEN 根因是… | 立即行动 |
|-------------|------------|---------|
| 仪表盘没人看 | 数据不相关或太难理解 | 从OMTT开始，只展示5个以内指标 |
| 数据延迟超过1天 | 数据管道未自动化 | 投入ETL自动化，确保T+1出数 |
| 不同人看到不同数字 | 数据口径不统一 | 建立指标字典，统一计算逻辑 |

### ⏸ CHECKPOINT-5：数据基础设施
> **通过标准**：仪表盘已上线，团队每周例会基于它做决策
> **自检**：如果仪表盘明天消失，团队能继续工作吗？能 → 数据已融入文化

## 技能8：数据驱动的A/B测试方法论

### 适用场景
任何需要验证假设、优化指标的场景

### 操作流程
1. **提出假设**：如果做X，指标Y会提升Z%
2. **确定样本量**：使用样本量计算器，确保统计显著性
3. **设计实验**：控制变量，只改变一个因素
4. **运行实验**：至少运行1~2个完整周期（避免周末效应等）
5. **分析结果**：看统计显著性（p < 0.05）
6. **做出决策**：赢→全量发布，输→学到洞察→下一个实验

### 常见陷阱
| 陷阱 | 说明 | 解决方案 |
|------|------|---------|
| 提前停止 | 数据一好看就停 | 预先确定样本量和运行时间 |
| 多重比较 | 测太多变体碰运气 | 限制同时测试的变体数 |
| 新奇效应 | 新版本初期数据好 | 等数据稳定后再判断 |
| 样本污染 | 用户被分到多个实验 | 确保互斥分组 |
| 忽略长期影响 | 短期提升但长期有害 | 追踪长期留存和LTV |

### 实验节奏
- 每周至少运行 2~3 个实验
- 维护实验想法库（保持 100+ 想法）
- 用 ICE 评分排序：Impact（影响力）× Confidence（信心）× Ease（容易度）

### ❌ 失败模式（IF-THEN）
| IF 你观察到… | THEN 根因是… | 立即行动 |
|-------------|------------|---------|
| 测试跑了2周p值还不显著 | 样本量不足或效果太小 | 先算需要多少样本，不够就继续跑或放弃 |
| 所有测试都"赢" | 要么测试太保守，要么多重比较问题 | 检查是否有同期多个实验互相干扰 |
| 测试赢了但上线后指标回落 | 新奇效应或样本偏差 | 上线后继续追踪1-2周确认 |
| 团队不做测试直接上线 | 文化问题/赶进度 | 建立"无数据不发布"的铁律 |

### 🚫 反模式黑名单
- ❌ **"直觉驱动"**：老板说改就改 = 没有验证，可能改坏
- ❌ **"事后解释"**：数据好了找理由，数据差了找借口 = 自我欺骗
- ❌ **"测试疲劳"**：连续测10个都无显著差异 → 不是测试没用，是假设质量差

### ⏸ CHECKPOINT-6：实验能力
> **通过标准**：团队能独立完成"假设→实验→分析→决策"全流程
> **自检**：过去30天做了几个实验？学到什么？0个 → 流程或文化有问题

## 技能9：Sean Ellis 测试 — 快速检验PMF

### 适用场景
判断产品是否达到产品-市场匹配（PMF）

### 操作方法
1. 向活跃用户发送问卷
2. 问一个问题：**"如果明天你不能再使用这个产品，你会有多失望？"**
   - 非常失望
   - 有些失望
   - 不太失望
   - 不适用——我已经不用了
3. 统计回答"非常失望"的比例

### 判断标准
- **≥ 40%** 回答"非常失望" → 你已经找到了PMF
- **25~40%** → 接近PMF，继续迭代
- **< 25%** → 还没找到PMF，需要调整方向

### 补充问题
问卷中还可以追问：
- 你从这个产品中获得的主要价值是什么？
- 你希望我们增加什么功能？
- 你会向谁推荐这个产品？（帮你定义目标用户画像）

### ❌ 失败模式（IF-THEN）
| IF 你观察到… | THEN 根因是… | 立即行动 |
|-------------|------------|---------|
| 得分<25%但团队觉得产品很好 | 自嗨，没解决真实痛点 | 做10个用户深度访谈，找到真实需求 |
| 样本量<50 | 统计不可靠 | 扩大样本，至少收到100份有效问卷 |
| 只问了新用户 | 新用户有新鲜感偏差 | 只问使用30天以上的活跃用户 |

## 技能10：创业阶段决策树

### 适用场景
面对多个增长机会时，决定优先做什么

### 决策树

```
你的留存曲线是否趋于平缓？
├── 否 → 回到"黏性"阶段
│   ├── 核心功能使用率低？→ 砍掉多余功能，聚焦核心价值
│   ├── 新用户流失严重？→ 优化新用户引导
│   └── 用户不知道怎么用？→ 改善教育和引导
│
└── 是 → 留存稳定，可以考虑增长
    ├── 病毒系数 > 0.5？
    │   ├── 是 → 继续优化传播链
    │   └── 否 → 增加分享触发点、简化分享流程
    │
    ├── LTV/CAC ≥ 3？
    │   ├── 是 → 可以扩大获客投入
    │   └── 否 → 先优化单位经济效益
    │       ├── LTV太低？→ 提高客单价/留存
    │       └── CAC太高？→ 优化渠道/转化率
    │
    └── 是否已准备好规模化？
        ├── 是 → 扩展新市场/渠道
        └── 否 → 继续深耕现有市场
```

## 技能11：构建精益数据文化

### 适用场景
从"拍脑袋"决策转向数据驱动决策的组织转型

### 文化建设清单

**第一阶段：基础设施**
- [ ] 部署基础数据采集工具（GA、Mixpanel等）
- [ ] 建立数据看板，全团队可见
- [ ] 定义核心指标和及格线

**第二阶段：流程建立**
- [ ] 每周数据例会（30分钟：看OMTT → 回顾实验 → 决定下周实验）
- [ ] 实验想法库（全员可提交）
- [ ] 实验记录模板（假设、设计、结果、学到什么）

**第三阶段：能力提升**
- [ ] 全员数据素养培训（读懂图表、理解统计显著性）
- [ ] 产品/设计/工程师能独立跑基础分析
- [ ] 建立数据驱动的决策习惯（"让我们用数据说话"）

**第四阶段：文化固化**
- [ ] 决策必须附带数据支持
- [ ] 失败实验和成功实验同等重要（都要记录和分享）
- [ ] 定期校准OMTT和及格线

### 避免的误区
- 不要让数据团队成为"守门人"——每个人应该能自己看数据
- 不要只看数据不看用户——定量分析 + 定性洞察 = 完整画面
- 不要追求完美数据——80%的确定性就足以做决策
- 不要数据独裁——数据是输入，不是唯一决策者
